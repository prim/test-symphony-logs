## 背景

当前 prepare 阶段里，GDB Server 的二进制路径已经统一走 `ResolveGDBBin()`，优先级为：

1. `MAZE_GDB_BIN`
2. `./gdb/prim-gdb/gdb-portable`
3. 系统 `gdb`

但 `prepare/corefile.go` 仍然直接调用系统 `gcore`：

```go
cmd := exec.Command("gcore", buildGcoreArgs(pid)...)
```

这会导致 core dump 抓取链路与 GDB Server 使用两套不同的调试器来源，也要求运行环境必须额外提供系统 `gcore`。

## 目标

不再依赖系统上的 `gcore`，统一通过 resolved GDB 生成 core dump，默认优先使用 `./gdb/prim-gdb/gdb-portable`。

## 方案

- 保留 `ctx.Corefile.Gcore(args.Pid)` 这一调用接口不变
- 在 `prepare/corefile.go` 内部去掉 `exec.Command("gcore", ...)`
- 改为使用 `ResolveGDBBin()` 获取 GDB 路径
- 通过 GDB 执行 `attach + generate-core-file/gcore + detach + quit`
- core 文件默认直接写入 `ctx.DirPath/core.<pid>`
- 继续兼容复用历史已有的 `core.<pid>` 文件，避免重复抓 core

- `prepare/gdb_server.go`
