## Verification Task

Verify the implementation for:

**prim/maze#138**: perf: optimize common mpm v2 hot paths

## 背景

`common/mpm/v2` 已经完成了相对 `v1` 的第一轮结构性重构，当前方向总体正确：

- 从固定大数组切换到只存非空页的 `map + pageIDs`
- `Collect` 阶段拆分跨页 piece，解决跨页查询 miss
- 用独立 `Known` 位图替代地址低位复用，已清理并发语义问题
- `Finalize` 后支持多轮 `IterPage` 扫描，和 Maze 的真实 workload 更匹配

结合现有调用模式分析、query-heavy benchmark 和 trace replay benchmark，可以判断 `v2` 的整体收益已经成立；下一阶段更适合围绕热路径继续做第二轮收敛，而不是重新设计一套新的 MPM。

参考 dev log:
- `dev-log/2026-03-13-mpm-v2-design-analysis.md`

## 目标

在保持 `mpm v2` 当前架构方向不变的前提下，继续压低热路径常数开销，优先优化：

1. finalize 后的 query 路径
2. `IterPage()` 调度开销
3. `Collect()` 阶段的中间分配

## 建议优化方向

### 1. 优化 `lookup()` 热路径

当前 `lookup()` 每次查询都会进行：

- `RLock/RUnlock`
- `map` 查找
- 页内二分

而 `Finalize()` 后 `pages` 和 `pageIDs` 已经稳定，真正变化的只有 `Known` 位图。

建议优先尝试：

- 明确 finalize 后禁止结构性修改
- 将 query 路径改成无锁读取
- 重新 benchmark `SizeAndKnow` / `SizeAndIsKnow`

可选进一步方案：

- 在 finalize 后把页存储冻结为 `sortedPageIDs + sortedPages`
- 用页级二分替代 `map` hash 查找，验证是否进一步降低 query 成本

### 2. 优化 `IterPage()` 调度模型

当前 `IterPage()` 每次调用都会：

- 创建 channel
- 启动 `runtime.NumCPU()` 个 worker
- 逐个发送 pageID
- 再发送退出哨兵

建议优先尝试：

- 用 `close(q)` 替代哨兵退出
- 增加 small-page fast path：页数很少时直接串行执行
- 根据 benchmark 再决定是否需要复用 worker pool

### 3. 减少 `Collect()` 的中间分配

当前 `Collect()` 先通过 `splitCrossPagePieces(batch)` 生成新的 batch，再统一写入页结构。

如果跨页 piece 占比很低，这会引入额外分配和复制。

建议改为：

- 单页 piece 直接写入目标页
- 仅跨页 piece 走拆分逻辑

### 4. 评估 page 内布局优化

当前页内结构为：

- `Pieces []MemoryPiece`
- `Known []uint32`

可后续评估：

- 是否改成 `begins[]/ends[]` 的 SoA 布局
- dense page 是否适合做更紧凑编码
- `Known` 位图是否需要做更 cache-friendly 的布局处理

### 5. 为 dense page 评估轻量辅助索引

当前页内查询完全依赖标准二分。可考虑在 `Finalize()` 为 dense page 构建轻量索引，例如：

- 每隔固定数量 piece 建一个 coarse entry
- 先粗定位，再在局部窗口二分/线扫

### 6. 评估 `ResetKnownBits()` 的增量重置策略

当前 `ResetKnownBits()` 会遍历所有非空页并逐字清零 known 位图。

可后续评估：

- 仅重置 touched pages
- 用 generation/epoch 语义替代显式全量清零

## 优先级建议

建议按以下顺序推进：

1. 去掉 finalize 后 query 路径上的 `RWMutex` 成本
2. 优化 `IterPage()` 调度模型，至少加入 small-page fast path
3. 减少 `Collect()` 中跨页拆分带来的中间分配
4. 评估 dense page 的辅助索引是否值得引入
5. 视 benchmark 结果决定是否重构 page 内存布局

- `SizeAndKnow` / `SizeAndIsKnow` 是否继续下降

7: 
8: 上一轮 `common/mpm/v2` 热路径优化在 `Known` 位图置位路径中使用了 `sync/atomic` 的 `OrUint32`。
9: 
10: 但仓库 `go.mod` 明确声明最低 Go 版本为 `1.18`，而该 API 不属于 Go 1.18 可用接口。这会导致：
11: 
12: - 源码实现与仓库声明的构建基线不一致
13: - 在低版本 Go 环境中出现构建失败或兼容性漂移
14: - 热路径优化虽然方向正确，但破坏了项目的可维护构建契约
15: 
16: ## 解决方案
17: 
18: 保留当前 `Known` 位图“原子置位并返回旧值”的语义，但将实现改回 Go 1.18 兼容写法：
19: 
20: - 新增 `orUint32(addr, bit)` 辅助函数
21: - 内部使用 `atomic.LoadUint32 + atomic.CompareAndSwapUint32` 自旋循环
22: - `SetKnown()` 与 `SwapKnown()` 统一复用该辅助函数
23: 
24: 这样既能保持并发安全与既有语义，也不会依赖高版本 Go 新增 API。
25: 
26: ## 主要改动
27: 
28: ### 1. `common/mpm/v2/mpm.go`
