**prim/maze#153**: Feature: 将 postman-db 目录迁移至 db

## 背景
当前项目将运行期产物、Postman 结果、ELF 缓存、RocksDB 索引以及部分调试副产物统一写入 `./postman-db/`。该命名已不能准确表达目录职责：目录内不仅包含 Postman 场景数据，还承载 standalone/prepare/GDB 相关中间产物，并且 Go 主流程、Python/GDB 脚本、测试、文档等多处都已与该目录耦合。

本需求目标是制定并实施一份迁移方案，将根目录下的 `./postman-db/` 统一迁移为 `./db/`，在保证兼容历史 prepare/legacy 逻辑的前提下，统一目录语义并减少后续维护成本。

## 技术方案
采用“常量抽象 + 路径分层迁移 + 兼容性验证 + 文档收口”的方案，避免一次性散改硬编码字符串带来的遗漏风险。

- Go 层优先统一公共常量与主流程输出路径，覆盖 prepare/process、maps、elf、core、refs RocksDB、日志与结果文件寻址。
- Python/GDB 工具链同步更新，避免 Go 输出到 `db/` 而脚本仍从 `postman-db/` 读取造成断链。
- prepare 兼容逻辑区分“当前输出路径”与“历史兼容输入路径”：新产物统一写入 `db/`，历史 `postman-db` 输入保留识别/迁移能力。
- 正式文档、调试说明、脚本示例更新到 `db/` 规则；历史 dev-log 作为历史记录可不批量回写。

## 任务拆分
- 抽象运行期数据库根目录常量，统一 `postman-db`/`db` 路径生成入口。

- 更新 Python/GDB 命令工具链中的 `./postman-db` 硬编码引用。
- 调整主程序结果收集与日志回收逻辑，确保 `db/` 体系下产物可正确读取。
- 修正测试与迁移验证用例：默认断言切换到 `db/`，同时保留旧版 `postman-db` 输入兼容测试。

- 执行构建、迁移测试和代表性场景验证，确认新目录规则下结果生成与读取完整可用。

## 风险
- 迁移跨 Go、Python、GDB、测试、文档多个层次，若只改一层容易产生运行期断链。
