# Feature: 将 .maze.py 全部重写为 Go prepare 包

## 背景与目标

### 背景

当前 Maze 项目采用**两阶段管道架构**：

```
阶段 1: Python (.maze.py)  →  .storage.json / .cpp.json  →  阶段 2: Go (.maze 二进制)
```

Python 层 (`.maze.py` + `maze-py/*.py` + `maze_*.py`) 共约 **4900 行 Python 代码**，负责所有预处理工作：
- 生成 core dump、解析 `/proc/pid/maps`
- 与 GDB 交互获取结构体偏移/sizeof/符号/vtable
- 检测语言运行时（Python/Lua/Node.js/C）和内存分配器（ptmalloc/tcmalloc/jemalloc/mimalloc）
- C++ DWARF 类型分析
- 汇总数据并输出 JSON 文件供 Go 分析引擎消费

这套架构存在以下问题：
1. **双语言维护成本** — 需要同时维护 Python 和 Go 两个技术栈
2. **进程间通信开销** — 通过 JSON 文件序列化/反序列化传递数据
3. **Python 2.7 兼容包袱** — 大量 Python 2/3 兼容代码
4. **部署复杂度** — 需要确保目标环境有 Python 和各种依赖
5. **builtins 注入反模式** — Python 层通过 `builtins` 注入实现全局单例共享，依赖关系完全隐式化

### 目标

将 `.maze.py` 及其所有相关 Python 模块**完全重写为 Go `prepare` 包**，实现：
1. **单一语言** — 整个 Maze 分析流程由 Go 统一实现（GDB 内部脚本除外）
2. **内存直传** — prepare 阶段直接填充 `common.S` 和 `common.Cpp` 全局变量，不再生成中间 JSON 文件
3. **入口统一** — Python `maze` 入口脚本的功能并入 `main.go`，最终只有一个 Go 二进制
4. **消除 Python 依赖** — 运行时不再依赖 Python 环境（`gdb-scripts/*.py` 保留，它们在 GDB 内部运行）

## 技术方案

### 整体架构

迁移后的架构：

```
main.go
 ├── parseArgs()                          ← 合并 Python maze 脚本的参数解析
 ├── prepare.Run()                        ← 原 .maze.py 全部逻辑

 │    ├── process.Init(pid)
 │    ├── gdb.GenerateDebugScript()
```

### 关键设计决策

#### 1. GDB 交互：Go 实现 GDB Socket Client

复用现有的 GDB Socket Server（`gdb/gdb-server/gdb_socket_server.py`，在 GDB 内部运行，必须保持 Python），Go 侧实现 TCP 客户端。

**通信协议**（已有，保持不变）：
- TCP 连接到 `127.0.0.1:9999`
- 帧格式：`<4字节大端序长度><UTF-8 JSON 数据>`
- 请求：`{"type": "command|eval|ping", "data": "..."}`
- 响应：`{"success": bool, "output|value": "...", "error": "..."}`

```go
// prepare/gdb_client.go
type GDBClient struct {
    conn    net.Conn
    host    string
    port    int
    timeout time.Duration
}

func (c *GDBClient) Execute(command string) (*GDBResponse, error)
func (c *GDBClient) ExecuteBatch(commands []string) ([]*GDBResponse, error)
func (c *GDBClient) Evaluate(expression string) (*GDBEvalResponse, error)
func (c *GDBClient) Ping() bool
```

#### 2. 外部命令调用：保持 exec.Command

所有外部命令（`readelf`、`nm`、`strings`、`gcore`、`md5sum`、`ldd`、`./bin/s3`、`./parse_dwarf`）暂不替换为 Go 原生实现，统一通过 `exec.Command` 调用。

```go
// prepare/cmd.go — 统一的外部命令执行封装
func RunCommand(name string, args ...string) (string, error)
func RunCommandSilent(name string, args ...string) error
func CheckOutput(name string, args ...string) (string, error)
```

#### 3. 数据传递：直接填充 common.S 和 common.Cpp

不再生成 `.storage.json` 和 `.cpp.json`，prepare 包直接操作 `common` 包的全局变量：

```go
// prepare 包直接 import common，填充数据
import . "maze/common"

func fillCommonVars(ctx *Context) {
    S.Process.Pid = ctx.Process.Pid
    S.Process.Type = ctx.Language
    S.Process.Mallocer = ctx.Process.Mallocer
    S.Process.Maps = ctx.Process.Maps

    S.Structs = ctx.GDB.Structs
    S.Symbols = ctx.Symbols.Symbols
    // ...
    Cpp.Vtables = ctx.Cpp.Vtables
    Cpp.DeepLoopStructs = ctx.Cpp.DeepLoopStructs
    // ...
}
```

## 约束与注意事项

### 技术约束

1. **GDB Scripts 必须保留 Python** — `gdb-scripts/` 下的脚本在 GDB 内部运行，使用 GDB Python API（`gdb.execute`、`gdb.parse_and_eval`），无法改写为 Go。包括：
   - `gdb-scripts/gdbcommand.py` — `offsets-of`、`maze-call-global-function` 等 GDB 自定义命令
   - `gdb-scripts/v8_heap_layout.py` — V8 堆布局导出
   - `gdb-scripts/az_v8_isolate.py` — Isolate 地址猜测
   - `gdb-scripts/az_node_init.py` — Node.js 初始化
   - `gdb/gdb-server/gdb_socket_server.py` — GDB Socket Server

2. **外部命令暂不替换** — `readelf`、`nm`、`strings`、`gcore` 等外部命令通过 `exec.Command` 调用，后续可逐步替换为 Go 原生实现（`debug/elf` 标准库等）。

3. **mimalloc 版本检测变更** — 原来通过 `ctypes.cdll.LoadLibrary` 调用 `mi_version()`，Go 中需要改为其他方式：
   - 方案 A: 通过 GDB `p mi_version()` 获取
   - 方案 B: 通过 `strings` 搜索版本标识
   - 方案 C: 通过 `nm` 查找 `mi_version` 后用 `readelf` 读取附近数据

4. **Python 2.7 兼容性不再需要** — Go 代码无需考虑 Python 2.7 兼容性。但 GDB Scripts 仍需兼容（GDB 内置的 Python 版本可能是 2.7 或 3.x）。

5. **`@once` 装饰器等价实现** — Go 中使用 `sync.Once` 替代 Python 的 `@once` 装饰器，确保初始化方法只执行一次。

### 性能考虑

1. **GDB 命令合并** — 原 Python 层逐条发送 GDB 命令，Go 实现时考虑批量发送（`execute_batch`）减少网络往返。
2. **并发初始化** — 无依赖关系的初始化步骤（如符号下载、版本检测）可以使用 goroutine 并发执行。
3. **缓存复用** — 保持现有的 `ctypes-{build_id}` 缓存机制，避免重复执行 GDB 命令。

### 迁移策略

1. **Phase 0-1 先行** — 完成基础设施和核心模块后，即可进行端到端验证
2. **双轨并行期** — 在完全迁移前，可以保留 `.maze.py`，通过环境变量或参数切换使用 Python 层还是 Go prepare
3. **逐步替换** — 每完成一个模块，运行回归测试确认无误
4. **最后删除** — 所有测试通过后，再执行 Phase 6 的清理工作
