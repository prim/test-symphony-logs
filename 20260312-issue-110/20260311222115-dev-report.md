**PARTIAL**

本轮已修复并验证最新 QA 报告中的运行时阻塞问题：`prepare.Run()` 之后进入统一分析主链路时，不再因 ptmalloc 所需元数据缺失/路径解析错误而在 `initPtmalloc sizeof _heap_info` 处 panic。Python `--tar`、C++ `--tar` 与 `postmanprofile` 主链路均已通过回归验证。

- `prepare/process.go`: 新增 `ResolveExistingPath`，在 tar/postman 场景下解析可访问真实文件路径
- `prepare/utils.go`: 新增 ELF 输入路径校验与统一解析工具
- `prepare/gdb.go`: GDB 命令/脚本改为使用可访问真实路径，并统一做路径转义
- `prepare/gdb_client.go`: GDB socket client 初始化命令增加路径转义
- `prepare/alloc.go`: allocator 符号定位前统一解析真实 ELF 路径
- `prepare/cpp_analyzer.go`: C++ 分析过程统一使用真实路径并修复 GDB `file` / `core-file` 调用
- `prepare/symbols.go`: 调试符号下载与符号定位统一使用真实 ELF 路径
- `python/dict.go`: 限制异常大基类 tuple 的遍历，并改用安全指针读取

## Self-Review 发现与修复
- 发现 tar/postman 场景下 maps 原始路径不可直接用于 `file`、`readelf`、`nm`、调试符号下载；已统一改为先解析当前环境可访问的真实 ELF 路径
- 发现 GDB 命令直接拼接路径在带特殊字符场景下存在风险；已统一增加路径转义
- 发现 prepare 后公共运行时状态初始化分散，容易遗漏 `sizeof_*` 与 mmap owner 初始化；已集中到 `initPreparedRuntime()`
- 发现 `readelf` stderr 会污染输出；已重定向到 `/dev/null`
- 发现 Python `tp_bases` 异常大 size 时存在不安全读取风险；已增加上限保护并改为安全指针读取
