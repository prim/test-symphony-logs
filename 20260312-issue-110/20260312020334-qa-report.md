## 失败的 Test Case

### TC-004B: Python→Go 对应模块核心逻辑已迁移
- **结果**: FAIL
- **预期行为**: `prepare.Run()` 不仅要有对应 Go 模块承接职责，还应在结果层面与 legacy 预处理输出保持一致。
- **实际行为**: `go test -count=1 ./qa/issue110 -v` 中 `TestPrepareRunMatchesLegacyJSONBaseline` 失败，出现明确结构差异：
  - Python case：`storage.deep_loop_structs.10000` 数组长度不一致，`30 != 6`
  - C++ case：`storage.deep_loop_structs.4618[0].pos` 数值不一致，`304 != 8`
- **复现步骤**:
  1. 在仓库根目录执行 `go test -count=1 ./qa/issue110 -v`
  2. 观察 `TestPrepareRunMatchesLegacyJSONBaseline` 输出
- **可能原因**: Go prepare 在深层结构采集、位置记录或对象展开策略上与 legacy Python 实现存在偏差。
- **建议修复**: 对照 legacy 产物逐字段比对 `deep_loop_structs` 等关键结构的生成逻辑，补齐或修正 prepare 填充规则。

### TC-007: Python 主流程回归
- **结果**: FAIL
- **预期行为**: `python3 testdata/run_test.py python/20260129-complex-types-311` 应通过，且对象统计满足 `validate.py` 断言。
- **实际行为**: validate 失败；预期 `pymempool` 数量大于 `50000`，实际约 `30585`。
- **复现步骤**:
  1. 在仓库根目录执行 `python3 testdata/run_test.py python/20260129-complex-types-311`
  2. 查看测试输出中的 `pymempool` 统计结果
- **可能原因**: Python 语言识别、pymalloc / pymempool 相关准备数据、对象归类或去重逻辑在迁移后发生退化。
- **建议修复**: 重点检查 Python prepare 链路中与 pymalloc、类型布局、对象计数相关的准备阶段数据填充，必要时与 legacy `.storage.json` 逐项核对。

### TC-009: --tar 单入口分析能力回归
- **结果**: FAIL
- **预期行为**: 通过 `run_test.py` 间接触发的 `maze --tar` 分析应在代表性回归用例上通过验证。
- **实际行为**: `--tar` 主链路可运行，但 Python 代表性回归未通过 validate。
- **复现步骤**:
  1. 在仓库根目录执行 `python3 testdata/run_test.py python/20260129-complex-types-311`
  2. 观察 `maze --tar` 分析结束后的 validate 结果
- **可能原因**: `--tar` 模式下 prepare 结果与 legacy 不一致，导致下游分析统计偏差。
- **建议修复**: 结合 AC1 的 JSON 基线对比结果，优先修复 prepare 输出一致性，再复测 `--tar` 主链路。
