## 背景
在 standalone 模式下执行 `./maze --build --run --pid <PID>` 时，分析流程本身已经完成，并能输出内存汇总与 WebUI 路径，但随后在 `HttpSetup()` 阶段触发 panic：`Post "/new": unsupported protocol scheme ""`。

从现象看，当前流程仍尝试向相对路径 `/new` 发起 HTTP POST，请求缺少完整的协议、主机与端口信息，说明远端协调接口与本地 WebUI 展示链路被混用。

## 现象
- standalone 本地分析已完成，但最终因初始化 WebUI 会话失败而异常退出
- `--run` / `--http` 场景体验不稳定
- tar/core/在线进程入口在 HTTP 初始化链路上可能存在相同风险

## 建议方案
- 拆分“本地展示服务”和“远端协调服务”两类 HTTP 语义，避免 standalone 仅做本地展示时强依赖远端 `/new`
- 在 `HttpSetup()` / `HttpJsonRequest()` 之前增加 base URL 解析与合法性校验，禁止把裸路径 `/new` 直接传给 `http.Post`
- standalone 且未配置远端服务地址时，走本地降级路径：跳过 `/new` 注册，本地生成或复用会话信息，并打印明确日志
- 将该类配置缺失从 panic 改为可诊断错误或可恢复降级
- 保持显式配置远端 base URL 时的现有 `/new` 协议兼容性
