## 失败的 Test Case

### TC-006: GDB 模块边界可独立工作
- **结果**: FAIL
- **预期行为**: `maze_py/gdb.py` 作为 GDB 交互封装边界，应能自洽地提供其模块内声明的方法，不应存在基础依赖缺失；至少其自身定义的路径解析逻辑应可正常执行。
- **实际行为**: `GDB._resolve_gdb_bin()` 调用即报 `NameError`，暴露出新模块内部实现不完整。
- **复现步骤**:
  1. 执行：
     ```bash
     python3 - <<'PY'
     import maze_py.gdb as g
     print(type(g.GDB).__name__)
     print(hasattr(g.GDB, '_resolve_gdb_bin'))
     print(g.GDB._resolve_gdb_bin())
     PY
     ```
  2. 观察输出。
- **可能原因**: `gdb.py` 从 `legacy.py` 复制/包装了部分 GDB 逻辑，但遗漏了 `os`、`sys` 等必要 import；同时实现仍高度依赖 `legacy.LegacyGDB`，迁移未完全闭合。
- **建议修复**: 使 `maze_py/gdb.py` 成为自洽的 GDB 边界模块：补齐缺失依赖，并确保模块内声明的方法可独立工作；若继续保留 `legacy.LegacyGDB` 作为底层实现，也应保证 `gdb.py` 自身暴露的接口没有死代码或未完成迁移的残留。
