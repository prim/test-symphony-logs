## 失败的 Test Case

### TC-001: facade 删除运行时 implementation 状态
- **结果**: FAIL
- **预期行为**: `common/vm` 不再维护当前 implementation 的运行时绑定状态。
- **实际行为**: 运行时状态仍保留在 `common/vm/compile_time.go`，包括 `implementation`、`implementationBinding`、`SelectImplementation()`、`HasImplementation()`、`dispatchInit()`、`dispatchClose()` 等。
- **复现步骤**:
  1. 打开 `common/vm/compile_time.go`
  2. 搜索 `implementationBinding`、`SelectImplementation(`、`dispatchInit(`、`dispatchClose(`
  3. 可见运行时绑定与生命周期分发表现仍存在
- **可能原因**: 为兼容既有测试与生命周期逻辑，开发将运行时状态整体迁移到了新的兼容文件，而非按需求彻底收敛为 compile-time-only 结构。
- **建议修复**: 重新核对 AC1 的边界，避免仅通过“换文件位置”保留原有运行时状态；至少应将非必要的运行时绑定/选择接口从正式实现路径中彻底移除或仅限测试专用构件。

### TC-006: `common/vm` 生命周期兼容回归
- **结果**: FAIL
- **预期行为**: `go test ./common/vm/...` 应全部通过，且 `SelectImplementation`/`Close` 相关兼容行为不能回归。
- **实际行为**:
  - `TestSelectImplementationOnlyChangesBindingBeforeInit` 中，`Close()` 后再次 `SelectImplementation(VMV3)` 未正确更新当前类型，得到 `VMNone`
  - `TestDispatchCloseClearsBindingState` 中，关闭后调用 `GetMemory()` 触发空指针 panic，而不是安全返回 `ok=false`
- **复现步骤**:
  1. 在仓库根目录执行 `go test ./common/vm/...`
  2. 观察上述两个失败用例
- **可能原因**:
  - `dispatchClose()` 将 `currentBoundVMType` 置为 `VMNone`，但后续选择/重绑定路径未完整恢复
  - facade wrapper 始终调用 `implGetMemory`，而关闭后该函数变量未被重置到 noop，导致测试注入场景下出现 nil 调用
- **建议修复**: 统一关闭态、重绑定态与测试兼容钩子的语义，确保关闭后 facade 对外行为始终安全且可重复初始化。
