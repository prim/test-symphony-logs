## Task

Implement the following GitHub Issue:

**prim/maze#162**: vm: make facade compile-time-only and remove runtime implementation state

# 背景

`common/vm` 已经移除了运行时切换与 `acquireSnapshot()` 热路径原子协议，但当前 facade 仍保留一层运行时状态与 dispatch：

- `boundImplementation`
- `currentVMType`
- `initialized`
- `currentImplementation()`
- `dispatchGetMemory*()` / `dispatchGetMemoryRange()`

这意味着每次 `GetMemory*` 调用仍需要：

1. 读取当前绑定实现
2. 做空值检查
3. 再间接调用具体实现

当前需求前提已经很明确：

- 不需要运行时切换 VM 版本
- 接受通过重新编译切换 VM 实现
- 目标是继续降低 VM 门面层的运行时开销

# 目标

把 `common/vm` 进一步收敛为“编译期唯一实现、近无状态”的门面结构，尽量接近 C 式条件编译的效果：

- 每个编译产物只包含一个 VM 实现
- facade 不再维护 implementation 的运行时状态
- `GetMemory*` 直接转发到编译期选中的实现
- 不依赖运行时 dispatch 选择具体实现

# 方案

1. 继续使用 Go build tags 选择唯一 VM 实现：
   - 默认实现
   - `vm_lru`
   - `vm_full`
   - `vm_v3`

2. 删除 facade 中剩余的运行时实现状态：
   - `implementationBinding`
   - `boundImplementation`
   - `currentImplementation()`
   - `SelectImplementation()`
   - `initialized`

3. 将 `GetMemory*` 收敛为 compile-time-only wrapper 或函数直绑，例如：

```go
func GetMemory(addr uint64) (uint64, bool) {
    return v3.GetMemory(addr)
}
```

或：

```go
var GetMemory = v3.GetMemory
```

4. 重新评估 `vm_dual`：
   - 删除
   - 或降级为特殊调试模式，不再作为常规支持路径

5. 同步修改 compat 层、测试和文档，明确 VM 版本切换只能通过重新编译完成。

# 关键点

- 方案收益不依赖“编译器一定内联 wrapper”
- 即使 wrapper 最终没有被内联，它仍然比当前的 runtime state + dispatch 更轻
- 如果编译器进一步内联，则属于额外收益

# 验收标准

- `common/vm` 不再维护当前 implementation 的运行时绑定状态
- `GetMemory*` 不再通过 `currentImplementation()` 或等价 dispatch 取实现
- `vm_dual` 被移除或降级为非常规模式
- facade 热路径中不再出现运行时 implementation state 管理热点
- `refs=v0` / `refs=v1` 场景下的 CPU profile 中，facade wrapper 成本进一步下降
