## Summary

PARTIAL：本轮修复了 VM V3 热路径的生命周期一致性与未对齐读取风险，复用了同布局 VM 以显著降低冷启动分配开销，并保持 Python / Node.js 主链路回归通过。
但正式验收仍未完成：`BenchmarkSequentialScan` 复测为 `3.351 ns/op`，仍高于 `< 3ns` 门槛；`BenchmarkCompareV3ColdStart` 中 V3 仍慢于 V2，AC4 / AC9 继续失败。

## Changes

- `common/vm/v3/vm.go`: 引入只读 `readState` 快照，避免读取路径与关闭/重建路径共享可变状态；为 linear span 记录对齐元数据；复用同 segment 布局的 VM 骨架并在 `Init()` 时重绑数据源，降低冷启动初始化成本。
- `common/vm/v3/v3_contract_test.go`: 调整 benchmark/test 安装运行态状态的方式，兼容新的 `readState` / `defaultState` 双状态模型。
- `common/utils.go`: 文本输出仍按 `--limit` 截断，但 JSON 输出保留完整结果，避免回归验证因 TopN 截断丢失类型。
- `nodejs/map_info.go`: 放宽匿名对象属性摘要的显示长度，减少 Node.js 回归校验中类型特征被截断的问题。

## Self-Review 发现与修复

- 发现 V3 读取热路径虽然通过 `defaultState` 原子发布，但关闭与重建路径仍与读取路径共享同一状态对象；本轮拆分为只读 `readState` 快照，保证关闭时先撤销读取入口，再释放旧 reader。
- 发现 linear span 快路径按“地址是否对齐”决定是否直接解引用，未显式结合底层缓冲区基址对齐；本轮改为预计算 `aligned8/aligned4`，未对齐时回退到 `binary.LittleEndian` 路径。
- 发现冷启动开销主要来自每次 `Init()` 重建 VM 结构；本轮增加同布局 VM 复用并重绑 source，显著降低冷启动成本，但仍未追平 V2。
