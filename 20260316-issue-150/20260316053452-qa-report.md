## 失败的 Test Case

### TC-005: V3 与 V2 对比基准存在且可执行
- **结果**: FAIL
- **预期行为**: compare benchmark 全部可执行，且 V3 在所有场景优于 V2。
- **实际行为**:
  1. `BenchmarkCompareV3SequentialScanPageTable` 已修复并优于 V2
  2. `BenchmarkCompareV3RandomAccessPageTable` 已优于 V2
  3. `BenchmarkCompareV3ConcurrentAccessPageTable` 仍慢于 V2
  4. `BenchmarkCompareV3ColdStart` 显著慢于 V2
- **复现步骤**:
  1. 在仓库根目录执行：
     ```bash
     go test -tags vm_v3 ./common/vm -run '^$' -bench 'BenchmarkCompareV3(GetMemory8|GetMemory4|SequentialScan|SequentialScanPageTable|RandomAccess|RandomAccessPageTable|ConcurrentAccess|ConcurrentAccessPageTable|GetMemoryRange|ColdStart)$' -benchtime=50ms -count=1
     ```
  2. 观察 `ConcurrentAccessPageTable` 与 `ColdStart` 子项结果。
- **可能原因**:
  - `common/vm/v3/vm.go` 的 `Init()` 仍通过 `prepareVM()` / `newRuntimeStateWithCloser()` / `defaultState.Store()` 构建运行态，冷启动路径开销明显高于 V2。
  - page-table 并发路径虽然已优化，但在 `segmentAt()` → `lookupSegment()` → `pageTable.Lookup()` / L1 cache 路径上仍存在额外成本，导致 `ConcurrentAccessPageTable` 场景下无法稳定领先。
- **建议修复**:
  - 针对冷启动 compare 单独剖析 `Init()` 路径，确认是否存在 VM 缓存复绑、state 发布、mmap/source 绑定等额外初始化成本。
  - 继续优化 page-table 并发读热路径，直到 compare 基准实测中 V3 稳定快于 V2。
