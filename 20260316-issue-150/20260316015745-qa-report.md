## 失败的 Test Case

### TC-005: V3 与 V2 对比基准存在且可执行
- **结果**: FAIL
- **预期行为**: `common/vm/bench_compare_test.go` 中 V3 vs V2 对比测试全部通过，且 V3 在所有场景优于 V2。
- **实际行为**: 冷启动场景中 V3 明显慢于 V2，未达到“所有场景优于 V2”。
- **复现步骤**:
  1. 在仓库根目录执行 `GOFLAGS='-tags=vm_v3' go test ./common/vm -run '^$' -bench 'BenchmarkCompareV3(GetMemory8|SequentialScan|RandomAccess|ConcurrentAccess|ColdStart)$' -benchmem -benchtime=40ms -count=1`
  2. 观察 `BenchmarkCompareV3ColdStart` 输出。
- **可能原因**: V3 初始化路径在冷启动阶段仍需建立更重的 runtime state / segment / mmap 相关结构，导致启动成本远高于 V2。
- **建议修复**: 优化 `common/vm/v3.Init()` 与 `newVM()` 冷启动路径，减少一次性初始化与大块映射准备成本，并重新验证 compare benchmark 全场景结论。

### TC-008: 性能目标门槛校验
- **结果**: FAIL
- **预期行为**: V3 顺序扫描延迟 `< 3ns`，随机访问 `< 4ns`，并发访问 `< 1.5ns`。
- **实际行为**: 顺序扫描为 `3.695 ns/op`，高于正式门槛；随机访问与并发访问满足要求。
- **复现步骤**:
  1. 在仓库根目录执行 `GOFLAGS='-tags=vm_v3' go test ./common/vm/v3 -run '^$' -bench 'Benchmark(RandomAccess|SequentialScan|ConcurrentAccess)$' -benchmem -benchtime=40ms -count=1`
  2. 观察 `BenchmarkSequentialScan` 输出。
- **可能原因**: 当前顺序扫描仍走 `GetMemory` 热路径，线性 span / segment cache 命中后仍有额外分发与边界判断成本，尚未稳定压到 `< 3ns`。
- **建议修复**: 继续优化顺序扫描热路径，重点检查 `linearSpan` 命中后的读取分支、dispatch 层开销，以及是否还能减少 segment lookup / 边界判断成本。
