## 失败的 Test Case

### TC-005: V3 与 V2 对比基准存在且可执行
- **结果**: FAIL
- **预期行为**: `common/vm/bench_compare_test.go` 中 V3 vs V2 对比测试全部通过，且 V3 在所有场景优于 V2。
- **实际行为**: `GetMemoryRange` 与 `ColdStart` 两类场景 V3 明显慢于 V2。
- **复现步骤**:
  1. 在仓库根目录执行：
     ```bash
     go test -tags vm_v3 ./common/vm -run '^$' -bench 'BenchmarkCompareV3(GetMemory8|GetMemory4|SequentialScan|SequentialScanPageTable|RandomAccess|RandomAccessPageTable|ConcurrentAccess|ConcurrentAccessPageTable|GetMemoryRange|ColdStart)$' -benchmem -count=1
     ```
  2. 查看 `BenchmarkCompareV3GetMemoryRange` 与 `BenchmarkCompareV3ColdStart` 输出。
- **可能原因**:
  - `v3.GetMemoryRange()` 为保证返回 detached copy，会进行 `cloneMemoryRange`，导致每次调用产生分配与拷贝开销。
  - 冷启动路径可能在页表/模板/线性映射准备阶段引入了较大固定成本。
  - `common/vm/bench_compare_test.go` 的 `TestV3ComparePerformanceGuard` 未覆盖这两类场景，导致回归未被守护测试阻止。
- **建议修复**:
  - 明确 AC9 是否包含 `GetMemoryRange` 与 `ColdStart`；若包含，应补齐真实优化并纳入性能守护。
  - 若设计上允许这两个场景不优于 V2，应同步修正规格与测试，不应宣称“所有场景优于 V2”。

### TC-008: 性能目标门槛校验
- **结果**: FAIL
- **预期行为**: 随机访问 < 4ns，顺序扫描 < 3ns，并发访问 < 1.5ns。
- **实际行为**: 随机访问 `4.093 ns/op` 超过门槛；其余两项通过。
- **复现步骤**:
  1. 执行：
     ```bash
     go test -tags vm_v3 ./common/vm/v3 -run '^$' -bench 'Benchmark(GetMemory8|GetMemory4|SequentialScan|SequentialScanPageTable|RandomAccess|RandomAccessPageTable|ConcurrentAccess|ConcurrentAccessPageTable|HotspotAccess|UnalignedAccess|CrossSegment|CacheMiss|GetMemorySafe|MixedWorkload|ColdStart|PythonObjectTraverse|V8HeapScan)$' -benchmem -count=1
     ```
  2. 查看 `BenchmarkRandomAccess` 输出。
- **可能原因**: 热路径常量开销仍偏高，当前优化只让 `GetMemory8` 达到 `3.905 ns/op`，但 `BenchmarkRandomAccess` 的地址模式更接近真实随机分布，页表/L1 cache miss 成本仍未完全压低。
- **建议修复**: 针对随机访问专门分析 `segmentAt()` / 页表子表 / cache miss 路径，避免仅依赖线性 span 的乐观路径成绩。

### 附加阻塞问题: `go test -tags vm_v3 ./common/vm/...` 整体仍失败
- **结果**: FAIL
- **预期行为**: V3 包与上层 VM 包测试整体通过。
- **实际行为**: `common/vm/v3.TestClosingStateDisablesFastReadPath` 失败，报错为“closing 状态下 GetMemory 不应继续命中旧 fast path”。
- **复现步骤**:
  1. 执行：
     ```bash
     go test -tags vm_v3 ./common/vm/...
     ```
  2. 观察 `common/vm/v3` 包失败。
- **可能原因**:
  - `common/vm/v3/vm.go` 中 `GetMemory()` 快路径开头直接读取 `readState.Load()`；当 `state.owner == nil` 时，条件 `owner == nil || (owner.closing.Load() == 0 && owner.closer == nil)` 会直接放行。
  - `TestClosingStateDisablesFastReadPath` 仅设置 `defaultState.Store(state)`，未刷新 `readState`；而全局 `readState` 可能残留旧 state，使 `GetMemory()` 命中陈旧快路径，绕过当前 `defaultState.closing`。
- **建议修复**:
  - 收紧 `readState.Load()` 的直接命中条件，避免 owner 为空时无条件走快路径。
  - 检查 `newRuntimeState()` / `defaultState.Store()` / 关闭流程是否需要同步发布新的 `readState`，确保 closing 状态能阻断旧快路径。
