# QA 验收报告

## QA Result: FAIL

## 总体结果

**[STATUS: FAIL]**

本轮针对 `prim/maze#150` 的 VM V3 高性能版本，按“静态核对 → 关键 Go 测试 → testdata 主链路回归 → 对比基准抽样”顺序执行验收。构建、build tag、无锁首访 race、V3 benchmark 覆盖、Python/Node.js 主链路回归均通过；但在实际执行 `common/vm/bench_compare_test.go` 的 V2/V3 对比基准时，发现 **V3 并未在所有已覆盖场景优于 V2**，与 AC9 / 需求文档的“V3 在所有场景优于 V2”不一致，因此总体判定为 **FAIL**。

## Findings

- 发现 1：`common/vm/bench_compare_test.go` 已不再对“V3 必须跑赢 V2”做自动化断言，只保留结果一致性守护；这与 `plan.md` 中 AC9 的硬性表述不一致，且会让“性能更优”无法被测试自动拦截。
- 发现 2：实际抽样执行 `go test -tags vm_v3 ./common/vm -run '^$' -bench 'BenchmarkCompareV3(GetMemory8|GetMemoryRangeCopied)$' -benchtime=100ms -count=1` 时，`BenchmarkCompareV3GetMemoryRangeCopied` 在多个 size 上 V3 逊于 V2，直接违反 AC9 “all scenarios”。
- 发现 3：已静态确认相对 `origin/master...HEAD` 未修改 `cmd/*.py`，符合最新 human review 强约束“不要改 cmd/*.py”。

## 失败的 Test Case

### TC-005: V3 与 V2 对比基准存在且可执行
- **结果**: FAIL
- **预期行为**: `common/vm/bench_compare_test.go` 中的 V3 vs V2 对比测试应证明 V3 在所有覆盖场景优于 V2，至少不应出现已覆盖场景的实测回退。
- **实际行为**: `BenchmarkCompareV3GetMemoryRangeCopied` 的 `size-16`、`size-1024`、`size-4096` 均出现 V3 慢于 V2。
- **复现步骤**:
  1. 在项目根目录执行 `./maze --build`
  2. 执行 `go test -tags vm_v3 ./common/vm -run '^$' -bench 'BenchmarkCompareV3(GetMemory8|GetMemoryRangeCopied)$' -benchtime=100ms -count=1`
  3. 对比 `v2-full` 与 `v3` 的 `ns/op`
- **可能原因**: 当前 compare guard 已改成结果一致性守护，不再对性能优胜做自动化断言；同时 V3 的 range copy 路径在部分尺寸下仍存在额外开销，未达到 AC9 所要求的“all scenarios 优于 V2”。
- **建议修复**: 由开发补充稳定且能反映真实目标场景的性能验收机制，并针对 `GetMemoryRange`/拷贝路径继续优化；在满足 AC9 前，不应将该项标记为通过。

### TC-008: 性能目标门槛校验
- **结果**: FAIL
- **预期行为**: V3 应稳定满足 issue 中的性能目标，并且对比基准不应出现明显回退样本。
- **实际行为**: 对比基准抽样已发现回退样本，导致性能验收缺乏完整正向证据链。
- **复现步骤**:
  1. 执行 `go test -tags vm_v3 ./common/vm -run '^$' -bench 'BenchmarkCompareV3(GetMemory8|GetMemoryRangeCopied)$' -benchtime=100ms -count=1`
  2. 观察 `BenchmarkCompareV3GetMemoryRangeCopied` 不同 size 的 `v2-full` 与 `v3` 结果
- **可能原因**: V3 随机读热路径已具备优势，但 range 读取/复制相关路径未同步优化，导致部分 benchmark 仍落后于 V2。
- **建议修复**: 将 AC3/AC4/AC5/AC9 的性能证据与具体 benchmark 场景一一绑定，并补充对回退场景的专项优化与回归门槛。
