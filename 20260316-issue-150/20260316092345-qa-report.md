## 失败的 Test Case

### TC-008: 性能目标门槛校验
- **结果**: FAIL
- **预期行为**: V3 随机访问延迟 `< 4ns`，顺序扫描 `< 3ns`，并发访问 `< 1.5ns`。
- **实际行为**: 顺序扫描与并发访问达标，但随机访问 benchmark 连续 3 次均在 `4.485 ~ 4.574 ns/op`；page-table 随机访问更慢，为 `13.39 ~ 13.48 ns/op`。
- **复现步骤**:
  1. 在仓库根目录执行 `./maze --build`
  2. 执行 `go test -tags vm_v3 ./common/vm/v3 -run '^$' -bench 'Benchmark(RandomAccess|SequentialScan|ConcurrentAccess)$' -benchmem -count=3`
  3. 观察 `BenchmarkRandomAccess` 的 `ns/op`
  4. 如需复核页表路径，再执行 `go test -tags vm_v3 ./common/vm/v3 -run '^$' -bench 'Benchmark(RandomAccessPageTable|SequentialScanPageTable|ConcurrentAccessPageTable)$' -benchmem -count=3`
- **可能原因**:
  - 当前 V3 快路径在随机访问场景下仍未达到 issue 设定阈值；
  - page-table 路径的随机访问成本明显偏高，说明 O(1) 映射实现虽可工作，但实际常数成本较大；
  - `bench_compare_test.go` 的守护测试更偏向“V3 不慢于 V2 某些场景”，不能替代 AC3 的绝对阈值校验。
- **建议修复**:
  - 重新剖析随机访问热路径，重点检查 `segmentAt()`、页表 lookup、L1 cache 命中与 `currentReadState()`/关闭态检查引入的额外常数开销；
  - 将 AC3 的绝对阈值断言固化到自动测试中，避免仅靠人工读取 benchmark 输出；
  - 单独优化 page-table 随机访问路径，否则“页表映射”主卖点与设计目标不一致。
