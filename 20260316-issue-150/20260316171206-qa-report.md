## 失败的 Test Case

### TC-005: V3 与 V2 对比基准存在且可执行
- **结果**: FAIL
- **预期行为**: `TestV3ComparePerformanceGuard` 通过，V3 在对比场景中稳定优于 V2。
- **实际行为**: `GetMemory8PageTable` 场景中 V3 持续慢于 V2，7/7 轮均未获胜。
- **复现步骤**:
  1. 在仓库根目录执行 `./maze --build`
  2. 执行 `go test -tags vm_v3 ./common/vm/...`
  3. 观察 `TestV3ComparePerformanceGuard` 的失败输出
- **可能原因**: 页表命中热路径仍有额外开销，当前 facade/segment/page table 的访问链路没有达到设计目标中的 hot path 优势。
- **建议修复**: 继续聚焦 `segmentAt`、页表 lookup 命中路径、以及 fast read 调度层开销，先让 `GetMemory8PageTable` 稳定优于 V2，再复验其余性能门槛。

### TC-008: 性能目标门槛校验
- **结果**: FAIL
- **预期行为**: AC3/AC4/AC5 对应的随机访问、顺序扫描、并发访问应达到 issue 设定阈值，并相对 V2 有明确提升。
- **实际行为**: 在最基础的页表读取对比守护未通过前，已无法证明整体性能目标成立。
- **复现步骤**:
  1. 执行 `go test -tags vm_v3 ./common/vm/...`
  2. 查看 `TestV3ComparePerformanceGuard` 的性能比较失败结果
- **可能原因**: 与 TC-005 相同，主热路径开销仍偏高。
- **建议修复**: 在修复 TC-005 后，再重新运行完整 benchmark 定标，补齐 AC3/AC4/AC5 的正式证据。

### TC-009: 关闭态 fast path 保护回归
- **结果**: FAIL
- **预期行为**: 进入 closing 状态后，`GetMemory` / `GetMemorySafe` / `GetMemory4` / `GetMemory4Safe` / `GetMemoryRange` 不应继续命中旧 fast path。
- **实际行为**: `TestClosingStateDisablesFastReadPath` 失败，说明至少 `GetMemory` 仍可在 closing 状态下读取旧 fast path。
- **复现步骤**:
  1. 执行 `go test -tags vm_v3 ./common/vm/v3 -run TestClosingStateDisablesFastReadPath -count=1`
  2. 观察测试失败信息
- **可能原因**: `acquirePublicReadState()` / `currentReadState()` 与 closing 位的协作仍不一致，存在旧 read state 被继续使用的窗口。
- **建议修复**: 重新审查 `defaultState`、`readState`、`closing`、`inflight` 的状态机，确保 closing 后所有公共读取入口都无法继续走旧 fast path。
