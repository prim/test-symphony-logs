## 失败的 Test Case

### TC-002: 无锁加载并发首访无竞态
- **结果**: FAIL
- **预期行为**: `go test -race` 通过且无 data race / runtime fatal，segment 首次并发访问只加载一次。
- **实际行为**: 执行 `TestConcurrentLazyLoadRaceFree` 时，Go 运行时报错 `fatal error: checkptr: pointer arithmetic result points to invalid allocation`，调用栈为：
  - `maze/common/vm/v3.(*Segment).fastBase` (`common/vm/v3/vm.go:330/346`)
  - `maze/common/vm/v3.(*VM).fastReadUint64` (`common/vm/v3/vm.go:1072`)
  - `maze/common/vm/v3.(*VM).GetMemory` (`common/vm/v3/batch.go:198`)
  - `maze/common/vm/v3.TestConcurrentLazyLoadRaceFree` (`common/vm/v3/v3_contract_test.go:127`)
- **复现步骤**:
  1. 在仓库根目录执行 `./maze --build`
  2. 执行 `go test -race -tags vm_v3 ./common/vm/v3 -run TestConcurrentLazyLoadRaceFree -count=1`
  3. 观察到 `checkptr` fatal，测试进程退出失败
- **可能原因**: `Segment.fastBase()` 将 `unsafe.SliceData(data.buf)` 转为 `uintptr` 存入 `basePtr`，再恢复为 `unsafe.Pointer` 供热点读路径使用；在 `-race` / `checkptr` 语义下，这种跨 GC/逃逸边界的裸指针缓存可能不再被视为有效分配对象指针，导致并发首访路径触发非法指针算术检查。
- **建议修复**: developer 应重点审查 `common/vm/v3/vm.go` 中 `basePtr` 缓存与 `fastBase()/fastReadUint64()/fastReadUint32()` 的 `unsafe.Pointer` / `uintptr` 互转策略，确保在 race/checkptr 下仍满足 Go 指针安全规则；必要时改为保留真实对象引用或采用不会跨分配边界失效的访问方式。
