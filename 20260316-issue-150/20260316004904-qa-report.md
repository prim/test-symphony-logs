## 失败的 Test Case

### TC-008: 性能目标门槛校验
- **结果**: FAIL
- **预期行为**:
  - 随机访问 `< 4ns`
  - 顺序扫描 `< 3ns`
  - 并发访问 `< 1.5ns`
- **实际行为**:
  - 随机访问 `4.831 ns/op`
  - 顺序扫描 `3.353 ns/op`
  - 并发访问 `0.2427 ns/op`
- **复现步骤**:
  1. 在仓库根目录执行 `./maze --build`
  2. 执行 `GOFLAGS='-tags=vm_v3' go test ./common/vm/v3 ./common/vm -run '^$' -bench 'Benchmark(SequentialScan|RandomAccess|ConcurrentAccess|CompareV3(GetMemory8|GetMemory4|SequentialScan|RandomAccess|ConcurrentAccess))$' -benchmem -benchtime=40ms -count=1`
  3. 观察 `BenchmarkSequentialScan` 与 `BenchmarkRandomAccess` 的 ns/op 指标
- **可能原因**: V3 当前已修复生命周期与 compare 守护问题，但随机访问/顺序扫描热路径仍存在额外常数成本；从代码看慢路径仍依赖 `segmentAt()`、页表/L1 缓存命中与跨段 fallback 组合，绝对门槛压缩尚未完成。
- **建议修复**: 继续针对 `GetMemory/GetMemory4` 与 `segmentAt()` 主路径做常数级优化，并在相同基准环境下复测 AC3/AC4。
