## 失败的 Test Case

### TC-005: V3 与 V2 对比基准存在且可执行
- **结果**: FAIL
- **预期行为**: compare benchmark 全部可执行，且 V3 在所有场景优于 V2。
- **实际行为**:
  1. `BenchmarkCompareV3SequentialScanPageTable` 在 `v2-full` 与 `v3` 子 benchmark 中均失败，错误为 `sequential read failed at 1400000`
  2. `BenchmarkCompareV3RandomAccessPageTable` 中 V3 慢于 V2
  3. `BenchmarkCompareV3ConcurrentAccessPageTable` 中 V3 慢于 V2
- **复现步骤**:
  1. 在仓库根目录执行：
     ```bash
     go test -tags vm_v3 ./common/vm -run '^$' -bench 'BenchmarkCompareV3(GetMemory8|GetMemory4|SequentialScan|SequentialScanPageTable|RandomAccess|RandomAccessPageTable|ConcurrentAccess|ConcurrentAccessPageTable|GetMemoryRange|ColdStart)$' -benchtime=50ms -count=1
     ```
  2. 观察 `BenchmarkCompareV3SequentialScanPageTable` 报错与 page-table compare 数值。
- **可能原因**:
  - `common/vm/bench_compare_test.go` 中 `benchmarkSequentialIteratorImplWithFixtures()` 使用 `startAddr := fixtures.baseAddr + 0x1000`、`endAddr := fixtures.segments[len(fixtures.segments)-1][1] - 8`，并通过 `addr += 8` 连续递增访问；在稀疏 segment 固件下会跨越 segment gap，导致访问到未映射地址，因而该 compare benchmark 本身失真并失败。
  - 即使忽略上述基准实现问题，同次实测中 page-table 随机/并发路径也未达到“V3 优于 V2”的要求，说明非线性路径仍存在真实性能短板。
- **建议修复**:
  - 修正 `BenchmarkCompareV3SequentialScanPageTable` 的访问模型，使其按真实 segment 边界推进，而不是跨 gap 线性递增。
  - 对 page-table 随机/并发路径继续优化，直到 compare benchmark 实测中 V3 全面优于 V2。
