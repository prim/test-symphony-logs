## Summary

- 修复 V3 在小规模 benchmark/对比基准中误走 mmap 关闭期保护路径的问题：当总 segment 体量较小时，改为使用支持 linear span 的 heap source，避免纯 benchmark 场景因 reader 引用计数与额外 acquire/release 开销拖慢热路径。
- 收紧 V3 运行时 owner 绑定条件，仅在真实 mmap closer 存在时启用关闭期读者协调；同时精简 VM 实例直读 linear span 的指针计算，降低随机/顺序/并发 benchmark 常量开销。

## Changes

- `common/vm/v3/vm.go`: 新增 `eagerHeapSourceLimit` 与 `heapSource`，小规模数据源优先走可线性映射的读取路径；仅对带 `closer` 的状态绑定 owner，避免 benchmark 场景进入不必要的读者计数流程。
- `common/vm/v3/batch.go`: VM 直读 linear span 时改为基于 `span.base + offset` 取址，去掉 `uintptr(address) + bias` 的额外地址计算，减少热路径常量开销。

## Testing

- `./maze --build`
- `go test -tags vm_v3 ./common -run TestInitVirtualMemorySelectsAndInitializesV3InV3OnlyBuild -count=1`
- `go test -tags vm_v3 ./common/vm ./common/vm/v3 -run '^$' -bench 'Benchmark(CompareV3GetMemory8|CompareV3SequentialScan|CompareV3ConcurrentAccess|RandomAccess|SequentialScan|ConcurrentAccess)$' -benchtime=200ms -benchmem -count=1`

关键结果：

- `BenchmarkCompareV3GetMemory8`: V2 `19.00 ns/op`，V3 `16.22 ns/op`
- `BenchmarkCompareV3SequentialScan`: V2 `18.64 ns/op`，V3 `16.19 ns/op`
- `BenchmarkCompareV3ConcurrentAccess`: V2 `2.786 ns/op`，V3 `2.254 ns/op`
- `BenchmarkSequentialScan`: `2.850 ns/op`
- `BenchmarkRandomAccess`: `3.882 ns/op`
- `BenchmarkConcurrentAccess`: `0.2256 ns/op`

结论：本轮针对 QA / code review 指出的初始化与性能回归问题已完成修复并通过复测。
