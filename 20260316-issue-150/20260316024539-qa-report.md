## 失败的 Test Case

### TC-005: V3 与 V2 对比基准存在且可执行
- **结果**: FAIL
- **预期行为**: `common/vm/bench_compare_test.go` 中 V3 在所有对比场景都优于 V2。
- **实际行为**: `ColdStart` 场景下 V3 为 `3306 ns/op`，慢于 V2 的 `1581 ns/op`。
- **复现步骤**:
  1. 在项目根目录执行 `./maze --build`
  2. 执行 `GOFLAGS='-tags=vm_v3' go test ./common/vm -run '^$' -bench 'BenchmarkCompareV3(GetMemory8|SequentialScan|RandomAccess|ConcurrentAccess|ColdStart)$' -benchmem -benchtime=80ms -count=1`
  3. 观察 `BenchmarkCompareV3ColdStart` 输出
- **可能原因**: V3 的 `Init()` / `prepareVM()` / `newRuntimeStateWithCloser()` / VM 复用冷启动路径仍有额外状态发布、缓存重绑或 source 绑定成本。
- **建议修复**: 重点分析 `Init()` 到首次 `GetMemory()` 的冷启动路径，减少 VM 复用时的状态切换与 source 绑定开销，并确认 compare guard 的判定与 issue AC 保持一致。

### TC-008: 性能目标门槛校验
- **结果**: FAIL
- **预期行为**: `BenchmarkSequentialScan < 3ns/op`，同时随机访问与并发访问满足 AC3/AC5。
- **实际行为**: `BenchmarkSequentialScan = 3.351 ns/op`，高于门槛；其余两项达标。
- **复现步骤**:
  1. 在项目根目录执行 `./maze --build`
  2. 执行 `GOFLAGS='-tags=vm_v3' go test ./common/vm/v3 -run '^$' -bench 'Benchmark(SequentialScan|RandomAccess|ConcurrentAccess)$' -benchmem -benchtime=80ms -count=1`
  3. 观察 `BenchmarkSequentialScan` 输出
- **可能原因**: 顺序扫描热路径仍经过 `readState.Load()`、边界判断与 fallback 分支判定，线性 span 快路径成本尚未压缩到目标范围。
- **建议修复**: 继续聚焦顺序扫描的快路径分支与状态访问成本，确认是否还能减少热路径中的边界检查、指针换算或 slow-path 切换概率。
