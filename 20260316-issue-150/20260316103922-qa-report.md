## 失败的 Test Case

### TC-008: 性能目标门槛校验
- **结果**: FAIL
- **预期行为**: V3 随机访问延迟稳定满足 AC3：`BenchmarkGetMemory8`、`BenchmarkGetMemory4`、`BenchmarkRandomAccess` 均应 `< 4ns`。
- **实际行为**: 本轮复测结果分别为 `4.034 ns/op`、`4.069 ns/op`、`4.072 ns/op`，均略高于门槛。
- **复现步骤**:
  1. 在项目根目录执行 `./maze --build`
  2. 执行：
     ```bash
     go test -tags vm_v3 ./common/vm/v3 -run '^$' -bench 'Benchmark(GetMemory8|GetMemory4|SequentialScan|SequentialScanPageTable|RandomAccess|RandomAccessPageTable|ConcurrentAccess|ConcurrentAccessPageTable|HotspotAccess|UnalignedAccess|CrossSegment|CacheMiss|GetMemorySafe|MixedWorkload|ColdStart|PythonObjectTraverse|V8HeapScan)$' -benchmem -count=1
     ```
  3. 观察 `BenchmarkGetMemory8`、`BenchmarkGetMemory4`、`BenchmarkRandomAccess` 的 ns/op 输出
- **可能原因**:
  - 随机访问快路径仍存在边际固定开销，优化余量不足，导致在当前机器上偶发越过 4ns 门槛
  - benchmark 对 CPU 频率、调度、缓存状态较敏感，当前实现缺乏足够安全裕量
  - 线性 span 与 page table 共存路径下的热点分支仍可能带来额外判断成本
- **建议修复**:
  - 继续压缩随机读取热路径分支与地址换算开销，确保在当前 CI / QA 机器上留出更明显余量，而非仅“贴线通过”
  - 如设计允许，可增加更稳定的性能守护策略，避免开发自测通过但 QA 独立复测超线
