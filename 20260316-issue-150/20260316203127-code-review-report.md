## 代码质量发现
- 发现 1：`switchImplementation()` / `dispatchInit()` 先执行新实现的 `Init()`，再发布新快照并等待旧实现 drain，这会让旧实现的在途读请求与新实现初始化阶段并发运行。由于 `Init()` 会修改 `shared` 包中的全局 hooks / segments / memory，并且各 VM 实现本身也依赖包级状态，这里缺少实现级隔离，存在切换期间读到被新实现重置过的共享状态的风险。该问题会影响运行时切换、重新初始化和关闭期稳定性，属于架构级生命周期缺陷。文件：`common/vm/facade.go:215-230`, `common/vm/facade.go:298-316`。严重程度：major
- 发现 2：`isPrivateMmap()` 从原先的强失败改为“打日志后返回 false”，把“进程映射信息缺失/失配”静默降级为“普通可读映射”。该函数被 `common/vm/v3/vm.go`、`common/vm/v2/vm.go` 以及多个 malloc 解析路径用于过滤保护页和分段边界；一旦 maps 不完整，当前实现会把本应视为保护区或未知区间的地址当成普通段继续参与模板构建和读取路径，容易把环境异常伪装成正常数据，给内存分析结果带来隐蔽污染。对于 Maze 这类内存布局工具，这种静默容错比显式失败更危险。文件：`common/vm/shared/helpers.go:358-365`。严重程度：major

## 建议改进
- 为 VM 切换建立明确的两阶段生命周期：先阻止旧快照进入、等待旧实现排空并完成关闭，再初始化新实现；如果必须保留“先初始化后切换”，则需要把 `shared` 状态和实现内部状态从包级单例改为实例级隔离。
- 对 `IsPrivateMmap` 的异常路径至少返回可被上层感知的错误/告警状态，避免将 maps 缺失直接等价为“非 private”；若必须降级，也应限制影响范围并增加显式统计/失败信号。
