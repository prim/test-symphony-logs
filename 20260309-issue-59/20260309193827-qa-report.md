## 失败的 Test Case

### TC-007: Node.js 输出类型名可正确格式化
- **结果**: FAIL
- **预期行为**: Node.js 结果中的一级类型名称应输出可读类型名或结构化未知类型名，不应直接出现 `formatResultType <id>` 这种内部 fallback 文本。
- **实际行为**: `maze-result.json` 中多项结果直接显示为 `formatResultType 72057594037928163` 等字符串。
- **复现步骤**:
  1. 执行 Node.js 回归（例如 `python3 testdata/run_test.py nodejs/20260225-maze-vs-heapsnapshot`）
  2. 查看生成的 `maze-result.json`
  3. 观察 `items[*].type`，可见多项显示为 `formatResultType <id>`
- **可能原因**:
  - `nodejs/maze.go` 的 `formatResultType` 未能覆盖全部 Node.js type id
  - fallback 直接把内部函数名样式字符串写入最终结果，未走统一结构化未知类型格式
- **建议修复**:
  - 补齐 Node.js type id 到可读名称的映射
  - 将未知类型 fallback 统一为可读、可诊断且面向用户的格式

### TC-008: 多组 testdata 回归执行要求
- **结果**: FAIL
- **预期行为**: 选取的 testdata 回归用例应具备完整测试资产，能直接通过 `python3 testdata/run_test.py <case>` 执行。
- **实际行为**: `nodejs/20260211-comprehensive` 缺少 tarball，无法启动完整回归。
- **复现步骤**:
  1. 在仓库根目录执行 `python3 testdata/run_test.py nodejs/20260211-comprehensive`
  2. 观察测试因目录内缺少 tar.gz 资产而失败
- **可能原因**: 该测试目录未按 `run_test.py` 的统一约定准备完整输入数据。
- **建议修复**: 补齐该 Node.js 用例所需 tarball，确保测试目录可被统一回归框架直接执行。
