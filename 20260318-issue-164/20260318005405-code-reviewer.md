# Refs v2 基于 BitalosDB 的后端演进方案

**日期**: 2026-03-17
**任务**: 基于现有 `common/refs/v1` 创建 `common/refs/v2`，保持 refs 门面 API 不变，但将底层持久化后端从 `RocksDB + grocksdb` 切换为 `BitalosDB`。

## 背景

当前 refs 后端现状如下：

- `common/refs/v0` 是纯内存版，不负责持久化。
- `common/refs/v1` 是默认版，负责正向/反向索引查询与落盘。
- `common/refs/v1/refs.go` 直接依赖 `github.com/linxGnu/grocksdb`，并把 DB 打开、批量写入、迭代查询、关闭都写死在实现内部。

从代码结构看，`refs v1` 目前有两个明显特点：

1. 业务语义已经稳定
   - `AddIndexes` / `AddReverseIndex` / `GetReferents` / `GetReferrers` / `Save` / `Check` 这套门面已经被 `common/refs` 和 `common/refs.go` 兼容层稳定消费。
2. 存储实现仍然强耦合
   - `txosRefDB` / `txosRRefDB` 直接使用 `*grocksdb.DB`
   - `GetReferents` / `GetReferrers` 直接依赖 RocksDB iterator
   - `writeTasks` / `writeReverseTasks` 直接构造 RocksDB `WriteBatch`
   - Hooks 中路径函数仍命名为 `RocksDBPath`

这使得 refs 虽然已经完成“版本包 + 门面层”重构，但 `v1` 仍是“接口稳定、存储内核固定”的状态。现在如果要尝试 `BitalosDB`，最稳妥的路径不是原地替换 `v1`，而是：

- 保留 `v1` 作为当前稳定版
- 基于 `v1` 派生 `v2`
- 在 `v2` 中仅替换存储后端与相关初始化/读写实现
- 门面 API、兼容层调用方式、上层业务行为保持不变

### 3. 存储接口迁移原则

虽然本轮目标是“基于 v1 创建 v2”，但不建议简单把 `grocksdb` 调用逐行替换成 `bitalosdb` 调用。更稳妥的方式是先在 `v2` 内部抽一层很薄的存储适配。

建议在 `common/refs/v2` 内部约束为三类动作：

1. 打开/关闭 DB
2. 批量写入 key/value
3. 按 key 前缀范围迭代读取

这样可以把 `refs` 的业务语义和具体 DB API 之间留出一层边界，避免 `v2` 再次演变成新的“存储实现硬编码包”。

### 4. API 映射思路

以下映射基于当前 `v1` 的使用方式和 `bitalosdb` 公开文档能力制定：

| `v1` 中当前行为 | `v2` 中建议实现 |
|------|------|
| `openDB(path)` 打开 `grocksdb.DB` | 打开 `bitalosdb.DB`，保留 `CreateIfMissing` 等初始化语义 |
| `WriteBatch.Put()` 批量写 key/value | 使用 `bitalosdb` 的 batch 写入能力 |
| `db.Write(wo, wb)` 提交 batch | 改为 `v2` 的统一 `writeBatch()` 包装函数 |
| `NewIterator(ro)` + `Seek` + `Next` | 改为 `bitalosdb` iterator，并显式设置 lower/upper bound 或等价范围控制 |
| `DisableWAL(true)` | 在 `v2` 中按 `bitalosdb` 的写入/flush 语义做等价配置；若没有完全对应项，则保持“分析产物可丢弃、优先吞吐”的目标 |
| `Close()` | 替换为 `bitalosdb` 的关闭流程 |

需要特别注意的不是 API 名字，而是下面这几个语义是否一致：

- 批量写入何时真正可见
- iterator 遍历范围是否能稳定复用当前 `rangeKeys()` 方案
- 写入顺序和相同 key 的覆盖语义是否与 `v1` 一致
- 关闭前是否需要额外 `Flush()`

## 主要改动

### 1. 新增 `refs v2` 版本包

- 复制 `v1` 的 refs 行为模型
- 存储内核切换到 `BitalosDB`
- `CurrentVersion()` 返回 `v2`

### 2. 维持当前 key/value 编码协议

- 继续保留 `dbKey(k, ts)`
- 继续保留 `encodeSingleUint64` / `encodeUint64Pair` / `encodeUint64Slice`
- 这样可以避免存储引擎替换和数据格式替换同时发生

这很重要，因为如果第一阶段同时改：

- key 编码
- reverse index 紧凑表示
- DB 引擎

那么出现问题时很难判断是哪个层级引入了回归。

### 3. 泛化 hooks 与日志命名

当前 `refs` 的路径注入仍命名为 `RocksDBPath`，这在 `v2` 下已经失真。

建议第一阶段采取兼容式收敛：

```go
type Hooks struct {
    ...
    RocksDBPath func(string) string
    RefsDBPath  func(string) string
}
```

选择顺序建议为：

- 优先用 `RefsDBPath`
- 若为空，再回退到 `RocksDBPath`

这样可以让：

- `v1` 不受影响
- `v2` 不再把新的 DB 后端硬塞进旧名字
- 上层调用点可以渐进迁移

### 4. 控制默认版本变更节奏

建议分两步走：

1. 第一步
   - 增加 `refs_v2`
   - 默认仍为 `v1`
   - 通过单测、回归测试、实际 PID 分析对比 `v1` / `v2`
2. 第二步
   - 如果 `v2` 稳定且收益明确，再考虑把 `select_default.go` 指向 `v2`

这样比“直接让默认版本切到 v2”风险小得多。

## 关键代码

建议 `v2` 在内部先收敛成下面这种结构：

```go
type kvDB interface {
    Close() error
}

func openDB(path string) *bitalosdb.DB
func writeIndexBatch(db *bitalosdb.DB, tasks []iTask) error
func writeReverseBatch(db *bitalosdb.DB, tasks []riTask) error
func rangeScan(db *bitalosdb.DB, addr uint64) []uint64
```

查询函数保持业务侧不变，只替换底层扫描实现：

```go
func GetReferents(addr uint64) []uint64 {
    if ret := referentsCache.Get(addr); ret != nil {
        return ret
    }
    ret := rangeScan(txosRefDB, addr)
    referentsCache.Append(addr, ret)
    return ret
}
```

核心原则：

- `refs` 业务逻辑继续围绕“地址 -> 多版本片段值”工作
- DB 切换只影响 `open / batch write / range scan / close`
- `AddIndexes`、`AddReverseIndex`、`Save`、`Check` 的业务语义不改

## 风险与注意事项

### 1. 不要把 `v2` 做成 `v1` 的原地替换

如果直接在 `v1` 上改 DB：

- 很难回归对比
- 老问题和新问题会混在一起
- 文档、build tag、测试边界都会变得模糊

### 2. 需要重新确认 iterator 范围语义

`v1` 依赖：

- `dbKey(addr, ts)`
- `rangeKeys(addr)`
- iterator 从 `key0` seek 到 `key1` 前结束

`v2` 必须确保这一读取模型仍然成立，否则 `GetReferents()` / `GetReferrers()` 会漏数据或串 key。

### 3. 需要显式验证 flush/close 语义

`v1` 当前通过 batch 写入并关闭 DB 完成生命周期。切到 `BitalosDB` 后，需要确认：

- `Save()` 结束后查询是否立即可见
- 程序退出前是否必须额外 flush
- 测试与手工分析场景是否存在“写入成功但未持久化”的窗口

### 4. 命名和兼容层要一次收敛

如果 `v2` 已经不是 RocksDB，但 hooks、日志、错误信息仍写着 `rocksdb`，后续调试会很混乱。至少应在 `v2` 内做到：

- 新日志使用 `refsdb` 或 `bitalosdb`
- 新增中性 hooks 名称
- 保留旧字段仅用于兼容
