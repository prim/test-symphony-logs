## 背景

当前仓库中的反向引用管理同时存在两类实现：

- `common/refs.go`：当前主实现，包含分片内存索引、异步 Channel、RocksDB 持久化，以及 `AddIndexes` / `GAddReverseIndex` / `GetReferrers` / `GetReferents` / `SaveRefIndex` / `CheckRefs` 等全局入口。
- `python/traverse.go` + `python/globals.go`：仍保留 Python 包内的历史轻量版反向索引，依赖 `ReverseIndex1` / `ReverseIndex2` / `ReverseIndex` 三段式结构，`BuildReverseIndex()` 直接耦合这套本地数据结构。

同时，`nodejs`、`txos`、`python` 对 refs 的接入方式也不统一：

- `nodejs` 主要调用 `GAddReverseIndex()` / `GetReferrers()` / `SaveRefIndex()` / `CheckRefs()`
- `txos` 主要调用 `AddReverseIndex()` / `AddIndexes()`，并保留了 `txos/refs_shim.go` 兼容层
- `python` 同时依赖包内实现和 `maze/common`

这导致几个问题：

- refs 的历史实现、当前实现、空壳占位函数混杂在一起，版本边界不清晰
- `common` 包暴露了过多 refs 细节，和 MPM 重构前的状态类似
- 各语言模块依赖路径不统一，后续很难继续演进轻量版 / 持久化版 / 新版本实现
