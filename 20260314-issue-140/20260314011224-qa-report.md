## 失败的 Test Case

### TC-003: common 仅保留 refs 兼容转发层
- **结果**: FAIL
- **预期行为**: `common` 包应仅保留单一、清晰、真实接线的 refs 兼容转发层；若保留 `common/refs/compatcommon`，则应成为 `common/refs.go` 的实际复用入口，避免两套并存的兼容面。
- **实际行为**: `common/refs.go` 直接依赖 `maze/common/refs`，`compatcommon` 目录仅保留未接入的平行兼容层。虽然两边现在大多转发到同一个 facade helper，但对外仍形成两套并列 API 面，维护者无法从生产调用链看出哪一个才是权威兼容边界。
- **复现步骤**:
  1. 阅读 `common/refs.go`，确认其直接 `import refs "maze/common/refs"` 并直接调用 `refs.InitCompat`、`refs.SyncCompatMode` 等。
  2. 阅读 `common/refs/compatcommon/compat.go`，确认其仍导出同职责的 `InitCompat`、`SetRefFunc`、`Python*` 等接口。
  3. 全仓库搜索 `maze/common/refs/compatcommon`，可见仅 QA 测试与注释提及，生产代码没有实际引用该兼容包。
- **可能原因**: 本轮修复把“逻辑真相源”收敛到了 facade helper，但没有把“兼容层边界”也收敛为单点生产入口，导致结构问题只修到一半。
- **建议修复**: 二选一收敛：
  - 要么让 `common/refs.go` 显式复用 `common/refs/compatcommon` 作为唯一兼容入口；
  - 要么删除 `compatcommon` 这一未接线平行层，并同步调整目录/测试/文档，明确兼容层唯一入口就是 `common/refs.go + maze/common/refs`。
