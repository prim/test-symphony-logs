## 失败的 Test Case

### TC-005: V3 与 V2 对比基准存在且可执行
- **结果**: FAIL
- **预期行为**: `common/vm/bench_compare_test.go` 中 V3 vs V2 对比测试全部通过，且 V3 在所有场景优于 V2。
- **实际行为**: `go test -tags vm_v3 ./common/vm/...` 中 `TestV3ComparePerformanceGuard` 失败，`RandomAccess` 场景出现 V3 不优于 V2。
- **复现步骤**:
  1. 在仓库根目录执行 `go test -tags vm_v3 ./common/vm/...`
  2. 观察 `maze/common/vm` 包测试输出
  3. 看到 `TestV3ComparePerformanceGuard` 在 `RandomAccess` 场景失败
- **可能原因**: 当前守护测试基于 wall-clock 直接比较，且 V3 热路径收益不足以稳定压过 V2；在边界条件接近时容易受调度、缓存状态、CPU 抖动影响。
- **建议修复**: developer 需要同时处理两类问题：一是继续优化 V3 单线程随机访问/顺序扫描热路径；二是重新审视 AC9 的测试方法，降低环境噪声导致的偶发误判，确保比较方法可重复、可稳定复现。

### TC-008: 性能目标门槛校验
- **结果**: FAIL
- **预期行为**: V3 随机访问 < 4ns、顺序扫描 < 3ns、并发访问 < 1.5ns，且相对 V2 达到对应提升比例。
- **实际行为**: 随机访问约 14.60ns/op，顺序扫描约 12.11ns/op，并发访问约 0.6340ns/op。
- **复现步骤**:
  1. 在仓库根目录执行 `go test -tags vm_v3 ./common/vm/v3 ./common/vm -run '^$' -bench 'Benchmark(RandomAccess|SequentialScan|ConcurrentAccess|CompareV3GetMemory8)' -benchmem -benchtime=20ms -count=1`
  2. 查看 `maze/common/vm/v3` 的 benchmark 输出
  3. 对照 issue #150 中 AC3/AC4/AC5 门槛
- **可能原因**: 当前 V3 虽已引入页表、L1 cache、lock-free loading 与 hot path 直接指针读取，但单线程随机访问和顺序扫描主路径仍未达到设计目标，可能仍受全局 hot state、lookup/slow path 回退、边界判断或当前测试模型与目标场景不一致影响。
- **建议修复**: developer 需要继续聚焦 `common/vm/v3/vm.go` 热路径和 lookup/cache 命中模型，优先解决 AC3/AC4，再复测 AC9。
