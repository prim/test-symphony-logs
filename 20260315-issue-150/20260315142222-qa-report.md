## 失败的 Test Case

### TC-005: V3 与 V2 对比基准存在且可执行
- **结果**: FAIL
- **预期行为**: `common/vm/bench_compare_test.go` 中所有 V3 vs V2 场景都应通过，且 V3 在所有场景优于 V2。
- **实际行为**: V3 在已执行的五个关键对比场景中全部慢于 V2。
- **复现步骤**:
  1. 在仓库根目录执行 `go test -tags vm_v3 ./common/vm -run TestV3CompareBenchmarksPresent -bench 'BenchmarkCompareV3(GetMemory8|GetMemory4|SequentialScan|RandomAccess|ConcurrentAccess)$' -benchmem -benchtime=20ms`
  2. 观察各子 benchmark 的 `ns/op`。
- **可能原因**: `common/vm/v3/pagetable.go` 在同一顶层页表槽位的多 segment 情况下未完整建立子页表 direct mapping，查询会回退到 `entry.start/end` 线性扫描；同时 `vm.go` 热路径固定成本仍较高。
- **建议修复**: 优先修正页表子结构的多 segment 建表逻辑，确保同一 1GB 桶内地址命中时可直接定位 segment，再继续针对 `lookupSegment`/读取主链路做热点优化。

### TC-008: 性能目标门槛校验
- **结果**: FAIL
- **预期行为**: 随机访问 `< 4ns`、顺序扫描 `< 3ns`、并发访问 `< 1.5ns`。
- **实际行为**: 随机访问 `21.11 ns/op`，顺序扫描 `18.22 ns/op`，仅并发访问 `0.7670 ns/op` 达标。
- **复现步骤**:
  1. 在仓库根目录执行 `go test -tags vm_v3 ./common/vm/v3 -run '^$' -bench 'Benchmark(RandomAccess|SequentialScan|ConcurrentAccess)$' -benchmem -benchtime=50ms`
  2. 记录输出中的 `ns/op`。
- **可能原因**: 页表命中路径和 segment 查找路径仍有明显固定开销；当前实现虽引入了 cache/page table，但未达到设计中的高性能 direct mapping 效果。
- **建议修复**: 在保证正确性的前提下继续压缩 `lookupSegment`、`segmentAt`、跨段读取与冷/热缓存路径上的分支和扫描成本，并补充更强的自动化阈值断言。
