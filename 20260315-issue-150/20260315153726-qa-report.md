## 失败的 Test Case

### TC-005: V3 与 V2 对比基准存在且可执行
- **结果**: FAIL
- **预期行为**: V3 在对比基准中优于 V2，且测试能够对不达标结果自动失败。
- **实际行为**: V3 在全部已执行场景中慢于 V2，benchmark 仅输出结果不做阈值断言。
- **复现步骤**:
  1. 在仓库根目录执行 `go test -tags vm_v3 ./common/vm -run '^$' -bench 'BenchmarkCompareV3(GetMemory8|GetMemory4|SequentialScan|RandomAccess|ConcurrentAccess)$' -benchmem -benchtime=20ms`
  2. 观察各子 benchmark 的 `v2-full` 与 `v3` 数值。
- **可能原因**: 当前 V3 热路径仍有额外段查找/边界处理/间接调用成本；同时对比测试没有建立性能门槛断言，导致 AC9 只停留在“可跑 benchmark”而不是“可验证优于 V2”。
- **建议修复**: 继续压缩读取热路径，并把 V3 优于 V2 的要求落到可失败的自动化检查中。

### TC-008: 性能目标门槛校验
- **结果**: FAIL
- **预期行为**: `RandomAccess < 4ns/op`，`SequentialScan < 3ns/op`，`ConcurrentAccess < 1.5ns/op`。
- **实际行为**: `RandomAccess = 14.66ns/op`，`SequentialScan = 12.06ns/op`，仅 `ConcurrentAccess = 0.5195ns/op` 达标。
- **复现步骤**:
  1. 在仓库根目录执行 `go test -tags vm_v3 ./common/vm/v3 -run '^$' -bench 'Benchmark(RandomAccess|SequentialScan|ConcurrentAccess)$' -benchmem -benchtime=20ms`
  2. 查看基准输出中的 `ns/op`。
- **可能原因**: 线性 span fast path 已做内联，但仍未把随机访问与顺序扫描降到目标区间；当前实现相对 V2 缺少足够强的直达路径优势。
- **建议修复**: 聚焦随机访问与顺序扫描的主读路径，继续减少 segment 定位与数据访问的常数开销，并在优化后重新做 V2/V3 对比验证。
