## 失败的 Test Case

### TC-005: V3 与 V2 对比基准存在且可执行
- **结果**: FAIL
- **预期行为**: `TestV3ComparePerformanceGuard` 通过，且 V3 在对比守护场景中稳定优于 V2。
- **实际行为**: `GetMemory8` 对比场景失败，V3 最佳结果仍慢于 V2。
- **复现步骤**:
  1. 在仓库根目录执行 `go test -tags vm_v3 ./common/vm -run TestV3ComparePerformanceGuard -count=1`
  2. 观察失败输出中的 `GetMemory8 性能未优于 V2`
- **可能原因**:
  - `bench_compare_test.go` 的守护测试使用 `testing.Benchmark` 多轮取 best-of-5，结果对热缓存、初始化路径、调度噪声较敏感。
  - `SwitchToV3()` 主链路下的 facade / shared hook / 初始化成本可能抵消了 V3 热路径收益。
  - V3 的真实快路径主要体现在 `hotSpan`/`hotSeg`，但 compare guard 的访问模式未稳定命中最佳路径。
- **建议修复**:
  - 收紧 compare guard 的对比口径，确保 V2/V3 使用完全一致且可复现的 shared hook、热身与访问模式。
  - 单独核对 `GetMemory8` compare benchmark 是否引入了额外初始化/切换噪声。
  - 在不改变语义的前提下，继续分析 V3 facade 入口与 hot path 的常数开销。
