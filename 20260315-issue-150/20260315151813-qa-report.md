## 失败的 Test Case

### TC-005: V3 与 V2 对比基准存在且可执行
- **结果**: FAIL
- **预期行为**: `common/vm/bench_compare_test.go` 中的 V3 vs V2 对比测试应证明 V3 在所有对比场景优于 V2，且具备可重复验证的强约束。
- **实际行为**: 当前仅验证基准函数存在；运行结果显示 V3 在全部已测对比场景中都慢于 V2。
- **复现步骤**:
  1. 在仓库根目录执行 `go test -tags vm_v3 ./common/vm -run TestV3CompareBenchmarksPresent -bench 'BenchmarkCompareV3(GetMemory8|GetMemory4|SequentialScan|RandomAccess|ConcurrentAccess)$' -benchmem -benchtime=20ms`
  2. 查看各子 benchmark 的 `ns/op`
  3. 对比 `v2-full` 与 `v3` 的结果
- **可能原因**: V3 通用热路径存在较高固定常数成本；此外 compare 测试当前更像“可运行性基准”，而不是“验收型性能断言测试”。
- **建议修复**: 由开发补充稳定的 V3 优于 V2 对比断言，并继续优化 `segmentAt/pageTable/ensureLoaded` 热路径常数开销。

### TC-008: 性能目标门槛校验
- **结果**: FAIL
- **预期行为**: 随机访问 < 4ns、顺序扫描 < 3ns、并发访问 < 1.5ns。
- **实际行为**: 随机访问 `15.40 ns/op`，顺序扫描 `12.33 ns/op`，仅并发访问 `0.5314 ns/op` 达标。
- **复现步骤**:
  1. 在仓库根目录执行 `go test -tags vm_v3 ./common/vm/v3 -run '^$' -bench 'Benchmark(GetMemory8|GetMemory4|SequentialScan|RandomAccess|ConcurrentAccess)$' -benchmem -benchtime=50ms`
  2. 检查 `BenchmarkSequentialScan` 与 `BenchmarkRandomAccess` 输出
- **可能原因**: 线性 fast path 虽做了预计算优化，但页表查找、segment 定位、原子加载可见性与跨路径分支判断仍带来较高固定成本。
- **建议修复**: 由开发继续聚焦热路径常数开销，尤其是 `segmentAt`、page table 命中后的访问链路，以及 compare benchmark 场景与真实优化路径的一致性。
