## 失败的 Test Case

### TC-005: V3 与 V2 对比基准存在且可执行
- **结果**: FAIL
- **预期行为**: `TestV3ComparePerformanceGuard` 通过，且 V3 在比较场景中优于 V2。
- **实际行为**: 守护测试失败，`GetMemory8` 场景下 V3 慢于 V2。
- **复现步骤**:
  1. 在仓库根目录执行 `go test -tags vm_v3 ./common/vm -run TestV3ComparePerformanceGuard -count=1`
  2. 观察测试失败输出
- **可能原因**: V3 虽已具备页表、L1 cache、linear span 等结构，但单线程热路径的地址定位/缓存命中/读取开销仍未压到低于 V2，导致关键比较守护测试失败。
- **建议修复**: 继续剖析 `GetMemory/GetMemory4` 单线程热路径，重点比较 `pageTable.Lookup`、segment cache 命中路径、linear span 快路径与 V2 full 模式的指令数量与分支成本。

### TC-008: 性能目标门槛校验
- **结果**: FAIL
- **预期行为**: 随机访问 `< 4ns`、顺序扫描 `< 3ns`、并发访问 `< 1.5ns`。
- **实际行为**: 仅并发访问达标；随机访问与顺序扫描分别为 `14.73 ns/op`、`12.15 ns/op`。
- **复现步骤**:
  1. 在仓库根目录执行 `go test -tags vm_v3 ./common/vm/v3 -run '^$' -bench 'Benchmark(RandomAccess|SequentialScan|ConcurrentAccess)$' -benchmem -benchtime=20ms -count=1`
  2. 查看 benchmark 输出中的 `ns/op`
- **可能原因**: 当前优化更偏向并发读场景；而单线程随机/顺序路径仍存在额外分支、segment 查询或内存布局访问成本，尚未达到 design doc 所要求的极限目标。
- **建议修复**: 对单线程关键路径做更细粒度 profiling，并与 V2 benchmark 同场对比，定位 `span hit`、`segmentAt`、fallback path 的具体耗时来源。
