## 失败的 Test Case

### TC-005: V3 与 V2 对比基准存在且可执行
- **结果**: FAIL
- **预期行为**: `common/vm/bench_compare_test.go` 中 V3 vs V2 对比测试全部通过，且 V3 在所有场景优于 V2
- **实际行为**: `TestV3ComparePerformanceGuard` 失败，首个失败场景即 `GetMemory8`，提示 `v2=32.064193ms v3=33.177583ms`
- **复现步骤**:
  1. 在仓库根目录执行 `go test -tags vm_v3 ./common/vm/...`
  2. 观察 `TestV3ComparePerformanceGuard`
  3. 可见 `GetMemory8 性能未优于 V2` 失败
- **可能原因**:
  - 从代码路径看，V3 仍存在 `segmentAt/pageTable.Lookup/L1 cache/hotSeg` 组合带来的额外常数项
  - 当前 `hotSeg` / `hotSpan` 为包级全局热点缓存，虽然并发场景快，但对单线程随机访问/顺序扫描的主链路优化仍不足以反超 V2
- **建议修复**:
  - 继续对 `GetMemory8/GetMemory4` 热路径做 profiling，优先减少慢路径分支与段定位成本
  - 复核 `pageTable.Lookup`、`segmentAt` 回退到 `sort.Search` 的条件与触发频率

### TC-008: 性能目标门槛校验
- **结果**: FAIL
- **预期行为**:
  - 随机访问 `< 4ns`
  - 顺序扫描 `< 3ns`
  - 并发访问 `< 1.5ns`
- **实际行为**:
  - 随机访问 `14.66 ns/op`
  - 顺序扫描 `12.04 ns/op`
  - 并发访问 `0.5320 ns/op`
- **复现步骤**:
  1. 在仓库根目录执行 `go test -tags vm_v3 ./common/vm/v3 -run '^$' -bench 'Benchmark(RandomAccess|SequentialScan|ConcurrentAccess)$' -benchmem -benchtime=20ms`
  2. 读取 benchmark 输出中的 ns/op
- **可能原因**:
  - 线性 span / hot segment 快路径尚未把常数项压到目标阈值内
  - 顺序扫描与随机访问场景可能仍频繁触发慢路径或缓存更新成本
- **建议修复**:
  - 对顺序扫描单独做连续地址访问优化，降低每次读取的段判定与缓存更新开销
  - 对随机访问补充更细粒度对比分析，定位 V2 更快的具体路径差异
