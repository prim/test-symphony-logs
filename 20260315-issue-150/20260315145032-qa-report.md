## 失败的 Test Case

### TC-005: V3 与 V2 对比基准存在且可执行
- **结果**: FAIL
- **预期行为**: `common/vm/bench_compare_test.go` 中的 V3 vs V2 对比测试全部通过，且 V3 在所有场景优于 V2。
- **实际行为**: 当前仅“基准存在且可执行”，但 V3 在所有已实现对比场景均慢于 V2。
- **复现步骤**:
  1. 在仓库根目录执行 `go test -tags vm_v3 ./common/vm -run TestV3CompareBenchmarksPresent -bench 'BenchmarkCompareV3(GetMemory8|GetMemory4|SequentialScan|RandomAccess|ConcurrentAccess)$' -benchmem -benchtime=20ms`
  2. 观察各子 benchmark 的 `v2-full` 与 `v3` 数值
  3. 可稳定复现 `v3 > v2`（数值更差）
- **可能原因**: 从现象看，V3 热路径上的 segment 定位、顺序扫描路径或额外间接层仍有明显常数开销；虽然页表与 cache 已接线，但整体访问链路尚未达到设计目标。
- **建议修复**: 继续针对 `lookupSegment`、顺序扫描场景、批量/单点读取热路径做剖析，并补充“为什么比 V2 慢”的分层 benchmark，定位是哪一层引入主要常数成本。

### TC-008: 性能目标门槛校验
- **结果**: FAIL
- **预期行为**: 随机访问 `< 4ns`，顺序扫描 `< 3ns`，并发访问 `< 1.5ns`。
- **实际行为**: 并发访问通过，但随机访问与顺序扫描均显著高于阈值。
- **复现步骤**:
  1. 在仓库根目录执行 `go test -tags vm_v3 ./common/vm/v3 -run '^$' -bench 'Benchmark(RandomAccess|SequentialScan|ConcurrentAccess)$' -benchmem -benchtime=50ms`
  2. 查看输出中的 `ns/op`
  3. 可复现 `RandomAccess ≈ 21.39ns/op`、`SequentialScan ≈ 18.27ns/op`
- **可能原因**: 当前实现的 O(1) 页表映射与 L1 cache 虽然存在，但没有形成对顺序与随机单点读取足够低的热路径成本；另外 `segmentAt` / `segmentIndexAt` / `ensureLoaded` / 跨 segment 处理中的分支与间接调用仍可能抬高常数时间。
- **建议修复**: 结合 CPU profile 或更细粒度 micro-benchmark，拆解页表命中、cache 命中、segment 加载后单次读取、顺序扫描等子阶段，定位瓶颈后再优化。
