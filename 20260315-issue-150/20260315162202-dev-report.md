## Summary

本轮针对 QA 报告中指出的 VM V3 热路径性能回退进行了最小化修复尝试，补充了 segment 级热点缓存与更直接的指针读取路径，并修正了线性 span 指针类型适配。相关改动可以明显缩小 V3 与 V2 在 `GetMemory8` 主链路上的差距，但尚未让 V3 在 compare guard 与 AC3/AC4 指标上全面达标。

结论：**PARTIAL**。本轮完成了局部热路径优化与稳定性修复，但 AC3 / AC4 / AC9 仍未满足，不能宣称全部完成。

## Changes

- `common/vm/v3/vm.go`: 增加 segment 级热点缓存字段；为单段热点访问补充快速路径；将线性 span 基址改为 `unsafe.Pointer` 并集中封装基址读取；抽出慢路径函数，减少热路径重复逻辑。
- `common/vm/v3/batch.go`: 复用统一的基址读取辅助函数，适配 `unsafe.Pointer` 基址类型。

## Self-Review 发现与修复

- 发现将基址类型从 `uintptr` 切换到 `unsafe.Pointer` 后，`batch.go` 的快速读取路径编译失败；已统一改为复用 `loadUint64Base/loadUint32Base`
- 发现新增快路径后并发懒加载测试曾出现瞬时失败；经修正基址处理并重复 10 次回归后恢复通过
- 发现本轮优化只能把 `GetMemory8` compare 差距从约 17ms 缩小到不足 1ms，但仍不足以翻转结论，因此不能判定问题已解决

## 注意事项

- 当前性能瓶颈已不再是明显的结构性大回退，而是热路径常数项仍略高于 V2；后续更可能需要减少 `segmentAt` / `pageTable.Lookup` / 缓存更新上的额外分支与原子成本
- 本轮遵守“仅修改源代码、不改测试”要求，未变更任何 `testdata/` 或 QA 文件
