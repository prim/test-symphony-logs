## 失败的 Test Case

### TC-005: V3 与 V2 对比基准存在且可执行
- **结果**: FAIL
- **预期行为**: `common/vm/bench_compare_test.go` 中的 V3 vs V2 对比测试通过，且 V3 在所有场景优于 V2。
- **实际行为**: `TestV3ComparePerformanceGuard` 失败，日志显示并发访问场景下 V3 慢于 V2。
- **复现步骤**:
  1. 在仓库根目录执行 `./maze --build`
  2. 执行 `go test -tags vm_v3 ./common/vm/...`
  3. 观察 `TestV3ComparePerformanceGuard` 失败输出
- **可能原因**: 当前 V3 热路径虽然引入全局 hot segment / linear span 快速读取，但在 `common/vm/bench_compare_test.go` 的 fixture 与调度模式下，仍存在额外路径切换、全局热点状态竞争或缓存命中不足，导致对比测试中某些场景不稳定落后于 V2。
- **建议修复**: 针对 `TestV3ComparePerformanceGuard` 的各子场景做 profiling，重点检查并发读路径中的全局热缓存共享、副作用写入，以及对比测试使用的访问模式是否绕开了当前优化路径。

### TC-008: 性能目标门槛校验
- **结果**: FAIL
- **预期行为**: 随机访问 < 4ns，顺序扫描 < 3ns，并发访问 < 1.5ns。
- **实际行为**: 随机访问 14.69ns，顺序扫描 12.04ns，仅并发访问 0.5402ns 达标。
- **复现步骤**:
  1. 在仓库根目录执行 `./maze --build`
  2. 执行 `go test -tags vm_v3 ./common/vm/v3 -run '^$' -bench 'Benchmark(RandomAccess|SequentialScan|ConcurrentAccess)$' -benchmem -benchtime=20ms -count=1`
  3. 查看 benchmark 输出
- **可能原因**: `GetMemory/GetMemory4` 虽做了 `unsafe.Pointer` 直接读取，但慢路径仍需要 segment lookup / cache 更新；同时全局 `hotSeg` / `hotSpan` 模式更偏向单一热点路径，未充分覆盖随机访问与 scan 基准的实际地址分布。
- **建议修复**: 继续收敛单线程热路径，检查是否还能减少地址范围判断、页表查询和热点状态写回；必要时补充更贴近 V2 的无写入只读快路径对照。
