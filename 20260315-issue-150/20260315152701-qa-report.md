## 失败的 Test Case

### TC-005: V3 与 V2 对比基准存在且可执行
- **结果**: FAIL
- **预期行为**: `common/vm/bench_compare_test.go` 中的 V3 vs V2 对比测试全部通过，且 V3 在所有场景优于 V2。
- **实际行为**: benchmark 虽然存在且可以运行，但 V3 在 GetMemory8 / GetMemory4 / SequentialScan / RandomAccess / ConcurrentAccess 五个主场景均慢于 V2。
- **复现步骤**:
  1. 执行 `go test -tags vm_v3 ./common/vm -run '^$' -bench 'BenchmarkCompareV3(GetMemory8|GetMemory4|SequentialScan|RandomAccess|ConcurrentAccess)$' -benchmem -benchtime=20ms`
  2. 对比输出中的 `v2-full` 与 `v3` ns/op 数值
  3. 观察到 `v3` 全部大于 `v2-full`
- **可能原因**: V3 页表 / 查找 / 懒加载热路径常数开销仍偏高；另外 compare benchmark 当前没有性能断言，只是输出结果，导致 AC9 缺少自动守护。
- **建议修复**: 继续压缩热路径分支与查找成本，并为 compare benchmark 增加稳定、可控的自动化性能判定或至少失败阈值检查。

### TC-008: 性能目标门槛校验
- **结果**: FAIL
- **预期行为**: V3 随机访问 `< 4ns`、顺序扫描 `< 3ns`、并发访问 `< 1.5ns`。
- **实际行为**: 随机访问 `14.60 ns/op`、顺序扫描 `12.10 ns/op`、并发访问 `0.5159 ns/op`。
- **复现步骤**:
  1. 执行 `go test -tags vm_v3 ./common/vm/v3 ./common/vm -run '^$' -bench 'Benchmark(RandomAccess|SequentialScan|ConcurrentAccess|CompareV3(GetMemory8|GetMemory4|SequentialScan|RandomAccess|ConcurrentAccess))$' -benchmem -benchtime=20ms`
  2. 读取 `maze/common/vm/v3` 包输出中的 `BenchmarkSequentialScan`、`BenchmarkRandomAccess`、`BenchmarkConcurrentAccess`
  3. 对照 issue #150 中的阈值判断
- **可能原因**: 当前线性 span 快速路径优化仍不足以覆盖主要热点，页表 + segment 查找在随机 / 顺序路径上的固定成本依然明显。
- **建议修复**: 优先针对随机访问和顺序扫描建立更短热路径，并在同一环境反复 benchmark 验证阈值是否真实可达。
