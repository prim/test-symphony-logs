## 失败的 Test Case

### TC-005: V3 与 V2 对比基准存在且可执行
- **结果**: FAIL
- **预期行为**: `common/vm/bench_compare_test.go` 中的 V3 vs V2 对比测试全部通过，且 V3 在所有主场景优于 V2
- **实际行为**: `TestV3ComparePerformanceGuard` 失败，提示 `GetMemory8` 场景下 V3 慢于 V2
- **复现步骤**:
  1. 在仓库根目录执行 `go test -tags vm_v3 ./common/vm/...`
  2. 观察 `TestV3ComparePerformanceGuard` 输出
  3. 可见失败信息：`GetMemory8 性能未优于 V2: v2=30.157792ms v3=48.511695ms`
- **可能原因**:
  - V3 虽已引入页表与 hot span 快路径，但随机/普通主链路仍存在额外查找与分发成本
  - 当前实现的线性 span 仅在满足连续映射条件时获得明显收益，通用访问路径尚未达到预期热路径开销
- **建议修复**:
  - 继续优化 `GetMemory/GetMemory4` 热路径，减少普通路径上的 segment 查找、fallback 判断与跨层调用开销
  - 在对比守护测试下逐项回归验证，确保不仅 benchmark 名称存在，而且真实结果全面优于 V2

### TC-008: 性能目标门槛校验
- **结果**: FAIL
- **预期行为**:
  - 随机访问 `< 4 ns/op`
  - 顺序扫描 `< 3 ns/op`
  - 并发访问 `< 1.5 ns/op`
- **实际行为**:
  - 随机访问 `14.53 ns/op`
  - 顺序扫描 `12.04 ns/op`
  - 并发访问 `0.5148 ns/op`
- **复现步骤**:
  1. 在仓库根目录执行：
     ```bash
     go test -tags vm_v3 ./common/vm/v3 -run '^$' -bench 'Benchmark(RandomAccess|SequentialScan|ConcurrentAccess)$' -benchmem -benchtime=20ms
     ```
  2. 查看输出中的 `ns/op`
- **可能原因**:
  - 顺序扫描与随机访问尚未充分利用设计目标中的 30%~50% 热路径收益
  - 当前页表/L1 缓存收益不足以抵消额外的结构与分支成本
- **建议修复**:
  - 对 `SequentialScan` 和 `RandomAccess` 进行专门 profiling，定位 page table lookup、cache miss、跨函数分发的主要耗时
  - 将性能守护值与 compare benchmark 同时作为回归门槛，避免局部优化后再次退化
