## 失败的 Test Case

### TC-002: 编译期 build tags 选择规则正确
- **结果**: FAIL
- **预期行为**: 当同时启用 `vm_lru` 与 `vm_full` build tags 时，应出现明确的编译期冲突，阻止产物生成。
- **实际行为**: `go test -tags 'vm_lru vm_full' ./common/vm/...` 成功通过，未产生任何编译错误。
- **复现步骤**:
  1. 在仓库根目录执行 `go test -tags 'vm_lru vm_full' ./common/vm/...`
  2. 观察 `maze/common/vm`、`maze/common/vm/shared`、`maze/common/vm/v1`、`maze/common/vm/v2` 均成功编译
  3. 未出现“双 tag 冲突”的编译失败
- **可能原因**: `common/vm/select_conflict.go` 目前只是声明了 `mustChooseExactlyOneVMBuildTag` 函数和一个空引用变量，这不会触发编译错误；它满足了“文件存在”和“包含标记”的静态检查，但没有形成真正的冲突保护。
- **建议修复**: 将冲突文件改为真正的编译失败机制，例如引入不可用标识、重复定义、非法常量表达式，或其他明确导致 `vm_lru && vm_full` 同开时构建失败的实现方式；修复后需要补充一个动态编译验证，避免再次仅通过静态字符串检查漏检。
