## Fix Task

QA or code review found issues with the implementation for:

**prim/maze#160**: vm: remove runtime switching from facade to eliminate acquireSnapshot hot path

# 背景

当前 `common/vm` facade 仍支持运行时切换 VM 实现，并通过 `acquireSnapshot()` / `releaseSnapshot()` 的读侧引用计数协议保护读路径。

在实际性能分析中，`refs=v0` 的主要热点不是 refs 本身，而是 VM 门面层的原子操作：

- `common/vm/facade.go:160` `snap.state.Add(1)`
- `common/vm/facade.go:176` `snap.state.Add(^uint64(0))`

这部分成本来自每次 `GetMemory*` / `GetMemoryRange` 调用都会经过 `acquireSnapshot()`，而 Python 分析阶段会高频调用这些接口。

当前需求已经明确：**不需要运行时切换 VM 版本**。

这意味着 VM facade 正在为一个当前不使用的能力付热路径成本。

# 目标

把 `common/vm` 从“支持运行时切换的动态门面”收敛为“初始化后固定实现的静态门面”，移除 `GetMemory*` 热路径上的 `atomic.Add` 开销。

# 方案

1. 删除或废弃运行时切换入口：
   - `SwitchToLRU()`
   - `SwitchToFull()`
   - `SwitchToV3()`
   - `CloseCurrent()`

2. 删除 snapshot 生命周期协议：
   - `implementationSnapshot`
   - `snapshotClosingBit`
   - `acquireSnapshot()` / `releaseSnapshot()`
   - `markSnapshotClosing()` / `waitSnapshotDrained()`

3. 将 `dispatchGetMemory*` / `dispatchGetMemoryRange` 收敛为直接调用当前绑定实现。

4. 收敛 `SetVMType()` 语义：
   - 仅初始化阶段生效
   - 不再表示运行时切换

5. 同步修改相关测试和文档，明确 VM 版本只在初始化阶段确定。

# 验收标准

- `common/vm/facade.go` 不再在 `GetMemory*` 热路径上执行 `atomic.Add`
- `go test ./common/vm/...` 通过
- 相关 QA / facade 契约测试更新并通过
- 对同一目标进程复测后，`pprof` 中 `acquireSnapshot` / `sync/atomic.(*Uint64).Add` 不再是全局热点
- `refs=v0` / `refs=v1` 的端到端耗时进一步收敛
