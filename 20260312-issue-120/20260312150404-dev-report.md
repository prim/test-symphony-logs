## 失败用例详细分析

### complex-types-35 和 complex-types-36

**根因**：Python 3.5/3.6 **不支持 dataclass**（Python 3.7+ 功能）。测试源程序 `complex_types.py` 中确实没有创建 Player 和 Config 类。但 `validate.py` 使用了 Python 3.11 版本的模板，包含了对 Player/Config dataclass 的校验。

**证据**：
- `complex_types.py` 注释明确写着："不使用 dataclass (Python 3.7+)"
- `complex_types.py` 中没有 `from dataclasses import dataclass`
- validate.py 第 113-125 行要求找到 Player 和 Config

**Maze 分析引擎行为正确**：pymempool_objects 已恢复到合理水平（py35=55970, py36=64154），PersonClass/GameEntity/TreeNode/SimpleClass 等类实例均正确识别。

**建议 QA 处理**：从 py35/py36 的 validate.py 中移除 "Validating dataclass instances" 部分（第 109-125 行），因为这些版本的测试程序没有创建 dataclass。

### comprehensive-types

**根因**：validate.py 的 `validate()` 函数签名为 `validate(json_file, py_merge=False)`，期望接收文件路径字符串。但 `run_test.py` 的调用方式是 `validate_module.validate(data)`，传入的是已解析的 JSON dict。

**建议 QA 处理**：修改 validate.py 的 `validate()` 签名为 `validate(data, py_merge=False)`，直接接收 JSON data dict，与其他所有 validate.py 保持一致。
