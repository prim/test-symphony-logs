## Findings

### 阻塞性问题：GDB `file` 命令使用了原始 maps 路径，导致无法加载调试符号（影响 AC1、AC3、AC4、AC5、AC6、AC8）

- **问题**: `prepare/gdb.go:626` 的 `InitServerForCorefile()` 向 GDB 发送 `"file " + ctx.Process.Exe`，使用的是 maps 文件中的原始路径（如 `/usr/local/bin/python3.11` 或 `/home/.../stack_multithread_test`）。在 tar 模式下，这些路径在本地不存在（文件已被 md5 重命名解压到临时目录），导致 GDB 无法加载可执行文件和调试符号。
- **后果链**:
  1. GDB `file` 命令加载失败 → GDB 没有调试符号信息
  2. `offsets-of struct _heap_info` 等命令返回空 → `Sizeof`/`Structs` 数据缺失
  3. `FillCommonVars()` 填充了空的 sizeof/structs 数据到 `common.S`
  4. Go 分析引擎因缺少关键数据 panic
- **证据 1 — Python 测试 panic**:
  ```
  panic: runtime error: index out of range [7] with length 0
  at python/dict.go:315 → PointerM(m, ...) 中 m 为空切片
  ```
  PyTypeObject 的结构体偏移可能不正确或缺失，导致计算出错误的内存地址。
- **证据 2 — C++ 测试 panic**:
  ```
  panic: initPtmalloc sizeof _heap_info
  at mallocer/ptmalloc/init.go:52
  ```
  `S.Sizeof["struct _heap_info"]` 为空，说明 GDB `Ctypes()` 没有成功获取 libc 结构体信息。
- **证据 3 — 两个测试均有 readelf 错误**:
  ```
  readelf：错误：无法找到程序解释器名称
  ```
  这表明 readelf 读取了不正确的文件（可能是 core dump 文件而非 ELF 可执行文件）。
- **根因分析**:
  - `process.go:initExe()` 将 `p.Exe` 设为 maps 中的原始路径（设计正确，保留原始信息）
  - 其他代码正确使用了 `Realpath()`: `gdb.go:167` (`GetBuildId`)、`symbols.go:15`、`process.go:347` (`setupLanguage`)
  - 但 `gdb.go:626` 和 `gdb.go:516` (GenerateDebugScript) 直接使用了 `ctx.Process.Exe` 而非 `ctx.Process.Realpath(ctx.Process.Exe)`
- **影响范围**: AC1, AC3, AC4, AC5, AC6, AC8 全部未通过
- **建议修复**:
  ```go
  // gdb.go:626 修改为:
  initCmds = append(initCmds, "file "+ctx.Process.Realpath(ctx.Process.Exe))

  // gdb.go:516 (GenerateDebugScript) 也需要同样修复
  ```

### 次要问题 1：`readelf` 错误消息 "无法找到程序解释器名称"

- **问题**: 两个测试输出中都有 `readelf：错误：无法找到程序解释器名称`
- **位置**: 推测来自 `process.go` 中的 `LoadOffset()` 调用 `readelf -a`，或 `Symbols.DownloadSymbolFiles()` 中的 `readelf -n`
- **影响**: 不确定是否导致数据缺失（可能只是 warning），但说明某些 readelf 调用仍然使用了错误路径或对 core 文件而非 ELF 文件调用
- **建议**: 检查所有 `readelf` 调用是否都使用了 `Realpath()` 解析后的路径

### 次要问题 2：`Libc.Address()` 中的冗余/无效代码

- **位置**: `alloc.go:165-174`
- **问题**: 回退逻辑中 `if _, ok := ctx.Symbols.Symbols["main_arena"]; !ok { if addr, ok2 := ctx.Symbols.Symbols["main_arena"]; ok2 { ... } }` — 从同一个 map 检查同一个 key，内层条件永远为 false。`mp_` 也是同样的问题。
- **影响**: 低，逻辑等价于无操作
- **建议**: 这段代码原意可能是从 GDB `info address` 的结果回退，但当前实现无效
