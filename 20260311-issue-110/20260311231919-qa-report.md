## 失败的 Test Case

### TC-010: 模式兼容与遗留依赖清理检查（聚焦 AC1 字段级一致性）
- **结果**: FAIL
- **预期行为**: `prepare.Run()` 生成并填充到 `common.S` / `common.Cpp` 的快照应与历史 `.storage.json` / `.cpp.json` 字段级一致。
- **实际行为**: 仓库级一致性测试失败，差异集中在 `deep_loop_structs` 字段：
  - Python 用例：`storage.deep_loop_structs.1 数组长度不同: 4 != 1`
  - C++ 用例：`storage.deep_loop_structs.1 仅存在于历史基线`
- **复现步骤**:
  1. 在仓库根目录执行 `go test -count=1 ./qa/issue110 -v`
  2. 观察 `TestPrepareRunMatchesLegacyJSONBaseline`
  3. 查看子用例 `python-complex-types-311` 与 `cpp-stack-multithread` 的报错
- **可能原因**:
  - `prepare/fill.go` 仅直接赋值 `Cpp.DeepLoopStructs = ctx.Cpp.DeepLoopStructs`，而历史 JSON 产物可能经过额外转换/归并逻辑。
  - `prepare/prepare.go` 中 `buildLegacyStorageSnapshot()` 通过结构体序列化再拼接 `Cpp`，但未必完全等价于历史 Python 版本对 `deep_loop_structs` 的组织方式。
  - 该差异同时出现在 Python 与 C++ 基线对比中，说明问题更可能位于 prepare → legacy snapshot 兼容层，或 C++ 分析结果填充逻辑，而不是单个语言用例特有数据。
- **建议修复**:
  - 逐字段对照历史 `.storage.json` / `.cpp.json` 生成逻辑，重点核对 `deep_loop_structs` 的结构、索引和值形态。
  - 修复后至少重新执行：
    - `go test -count=1 ./qa/issue110 -v`
    - `python3 testdata/run_test.py python/20260129-complex-types-311`
    - `python3 testdata/run_test.py cpp/20260304-stack-multithread`
