## 失败的 Test Case

### TC-004: .gitmodules 正确迁移 prim-gdb submodule
- **结果**: FAIL
- **预期行为**: `.gitmodules`、git metadata、submodule 初始化/同步配置应全部迁移到 `gdb/prim-gdb`，旧的 `prim-gdb` 配置不应残留。
- **实际行为**: git common config 中仍残留旧项：`submodule.prim-gdb.url`、`submodule.prim-gdb.active`；同时又存在新项：`submodule.gdb/prim-gdb.url`、`submodule.gdb/prim-gdb.active`。执行 `git submodule sync` 后旧项仍存在。
- **复现步骤**:
  1. 在当前仓库 worktree 中执行 `git rev-parse --git-common-dir`
  2. 记录返回的 common git 目录
  3. 执行 `git config --file "$(git rev-parse --git-common-dir)/config" --get-regexp '^submodule\.'`
  4. 观察输出中同时存在 `submodule.prim-gdb.*` 与 `submodule.gdb/prim-gdb.*`
  5. 执行 `git submodule sync`
  6. 再次执行第 3 步，旧配置仍然存在
- **可能原因**: 本次迁移更新了 `.gitmodules` 和 gitlink，但没有覆盖旧 submodule 名称在本地仓库元数据中的残留状态，导致配置清理不完整。
- **建议修复**: 由开发侧补充并验证 submodule 迁移后的元数据清理流程，确保从旧名称迁移到新名称时，本地 common config 与相关模块目录状态一致，并在测试中覆盖该场景。
