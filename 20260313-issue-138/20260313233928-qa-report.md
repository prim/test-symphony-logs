## 失败的 Test Case

### TC-006: Finalize 后并行查询微基准
- **结果**: FAIL
- **预期行为**: 根据需求与验收关注点，`SizeAndKnow` / `SizeAndIsKnow` 的 query-heavy 成本应继续下降，trace replay query 应继续优于当前基线或至少不给出反向信号。
- **实际行为**: `BenchmarkLookupAfterFinalizeParallel` 单点微基准很好，但更接近真实访问形态的 query-heavy 与 trace replay query 基准中，`V2` 的时间指标仍慢于 `V1`。
- **复现步骤**:
  1. 执行 `go test ./common -run ^$ -bench 'BenchmarkMPMSizeAndKnowHit|BenchmarkMPMSizeAndIsKnowMiss' -benchmem -count=1`
  2. 观察 `Sparse/Dense/Mixed` 各组 `V1` 与 `V2` 时间结果
  3. 执行 `go test ./common -run ^$ -bench 'BenchmarkMPMWorkloadMixedRounds|BenchmarkMPMTraceReplayQuery' -benchmem -count=1`
  4. 观察 `BenchmarkMPMTraceReplayQuery` 中 `V2` 时间仍高于 `V1`
- **可能原因**:
  - 当前“无锁查询”优化在极简单基准上收益明显，但在真实 query-heavy 数据形态下，`pageMap` 命中、页内二分与 `Known` 位图访问的综合成本仍偏高
  - 本轮优化主要解决锁与调度常数，但尚未兑现计划中提到的 dense page 辅助索引、页内布局进一步优化等潜在热点
- **建议修复**:
  - 继续围绕 query-heavy 场景做剖析，而不是只看 `BenchmarkLookupAfterFinalizeParallel`
  - 针对 dense page / replay query 增加剖面分析，评估页级辅助索引或页内布局优化是否必要

## 额外静态发现

- `common/mpm/v2/mpm.go` 中存在明显的接口语义不一致：
  - 包级 `SizeAndIsKnow()` 调用 `sizeAndIsKnowKnown()`，返回的是 **known bit**
  - 但方法级 `(*MemoryPieceManager).SizeAndIsKnow()` 在命中时固定返回 `true`，语义变成了 **是否找到对象**
- 这会导致同名 API 在不同调用入口上返回不同含义，属于高风险正确性问题。当前仓库内大多数直接调用 `m.SizeAndIsKnow()` 的地方没有消费第三返回值，因此暂未暴露，但该行为已经偏离接口命名与 facade 合约语义。
