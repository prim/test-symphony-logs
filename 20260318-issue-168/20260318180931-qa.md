**prim/maze#168**: Refactor local ResultItem aggregation to use AddResult consistently

## Background
Many upper-layer modules still hand-roll the same local ResultItem aggregation pattern:

- create local `map[uint64]*ResultItem`
- build `func(s uint64) *ResultItem` via `GetResultItem`
- run scan/classify logic
- call `MergeResultItems(resultItems)`

This logic is already encapsulated by `AddResult`, but usage is inconsistent across python / txos / cpp / mallocer paths.

## Goal
Unify upper-layer local aggregation to `AddResult(...)` where semantics are equivalent, while preserving existing batch boundaries and concurrency safety.

## Scope
- replace hand-written local ResultItem aggregation shells with `AddResult` in serial call sites
- for `IterPage` or worker-parallel code, use `AddResult` per page / per worker scope instead of wrapping the whole concurrent loop
- keep direct manual merge code unchanged where code constructs full `ResultItem` objects explicitly

## Constraints
- do not share one `AddResult` local map across multiple goroutines
- do not change side-effect order for logic that also updates reverse index / result object type maps

- verify no concurrent map write regression in `IterPage` paths
